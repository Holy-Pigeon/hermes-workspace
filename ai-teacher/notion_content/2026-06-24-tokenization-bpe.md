## 今日知识点：Tokenization 与 BPE（子词切分）——把一句话切成模型能吞下的最小单位

### 所属类目
类目 20：大语言模型应用

### 问题入口
Transformer 里所有运算——注意力、矩阵乘法、Softmax——吃的都是数字向量。可我们喂给 ChatGPT 的是一句话：「我爱学习 Transformer」。

那么第一个问题来了：**模型到底是怎么把这串文字变成它能处理的离散符号的？** 是按字母切？按单词切？按汉字切？这个看似琐碎的"切词"决定，其实深刻影响着模型的词表大小、能不能处理没见过的新词、甚至训练成本。今天我们就把这道"模型的第一道工序"讲透。

### 朴素想法
最自然的两种切法：

1. **按词切（word-level）**：把句子按空格/词典切成完整单词，每个词分一个编号。"I love cats" → ["I", "love", "cats"] → [5, 88, 1203]。
2. **按字符切（char-level）**：把句子拆成单个字符。"cats" → ["c", "a", "t", "s"]。词表极小（英文几十个，加标点上百个）。

这两种都很直觉，人脑读句子大概也是先看到一个个词。

### 卡住的地方
两种朴素办法各自撞墙：

**按词切的致命伤——词表爆炸 + OOV（Out-Of-Vocabulary，词表外词）**：
- 英语有几十万个词形（play, plays, playing, played, player...），中文加上各种组合更多。词表动辄几十万，而模型最后一层要对每个词算一个概率（Softmax），词表越大计算越贵、参数越多。
- 更糟的是**新词**：训练时没见过 "Transformer"、"GPT-5"、"奥利给"，模型只能把它标成一个统一的 `[UNK]`（unknown）符号，信息全丢。世界上的词是无穷的，词表却是有限的。

**按字符切的代价——序列太长、语义太碎**：
- "internationalization" 一个词被切成 20 个字符，序列长度暴涨。而注意力的计算复杂度是 $O(n^2)$（$n$ 是序列长度），序列一长，成本平方级飙升。
- 单个字符 "a"、"t" 几乎不携带语义，模型要从零拼出意思，学习负担重。

我们卡在一个两难：**词表小 vs 序列短，似乎不可兼得**。

### 1. 概念

**Tokenization（分词/词元化）**：把原始文本切成一串「token（词元）」，再把每个 token 映射到一个整数 ID 的过程。它是文本进入模型前的第一道工序。

**关键跃迁——子词切分（subword tokenization）**：不切到词，也不切到字符，而是切到**介于两者之间的"子词单元"**。常见词保留为整体，罕见词拆成更小的、有意义的片段。

**BPE（Byte-Pair Encoding，字节对编码）** 是最主流的子词算法之一。它的核心思想一句话：**从字符开始，反复把"最常一起出现的相邻一对"合并成一个新单元，直到达到目标词表大小。**

通俗理解：想象你在压缩一篇文章。你发现 "th" 这两个字母老是连在一起出现，于是发明一个新符号代替 "th"；接着发现 "the" 也很常见，再造一个符号……高频组合逐渐"长"成完整的词，低频词则保留为零碎片段。这正是 BPE 在做的事——它本来就源于一个**数据压缩**算法。但别停在"压缩"这个类比：BPE 的产物是一张「合并规则表」，它让常见词变成单个 token、罕见词优雅降级为几个子词，从而同时缓解了词表爆炸和 OOV。

形式化地，给定语料，BPE 维护一个词表 $V$（初始为所有基础字符）和一个有序的合并规则列表。每一步选出当前语料中频率最高的相邻符号对 $(a, b)$，记其出现次数为：

$$
(a^*, b^*) = \arg\max_{(a,b)} \text{count}(a, b)
$$

然后把 $a^*b^*$ 作为新符号加入 $V$，并记录这条合并规则。重复 $k$ 次，最终词表大小约为「基础字符数 $+ k$」。

### 2. 推导与证明

我们手动跑一遍 BPE 训练，体会它怎么"长出"词。假设语料里只有这几个词及其频率（词尾用 `</w>` 标记单词边界，这是经典 BPE 的常见做法）：

```
low </w>      : 5 次
lower </w>    : 2 次
newest </w>   : 6 次
widest </w>   : 3 次
```

**第 0 步：初始化为字符。**
词表 = {l, o, w, e, r, n, s, t, i, d, </w>}。每个词表示成字符序列，例如 `newest</w>` = `n e w e s t </w>`。

**第 1 步：统计所有相邻字符对的频率。**
- `e s` 出现在 `newest`(6) 和 `widest`(3) → 共 9 次，最高。

为什么选最高频对？因为合并最频繁的组合，能用最少的合并次数覆盖最多的文本——这是贪心的压缩直觉。

**第 2 步：合并 `e s` → `es`。**
现在 `newest` = `n e w es t </w>`，`widest` = `w i d es t </w>`。词表新增 `es`。

**第 3 步：再统计。**
- `es t` 出现 9 次（newest 6 + widest 3），最高 → 合并成 `est`。

**第 4 步：继续。**
- `est </w>` 出现 9 次 → 合并成 `est</w>`。注意此时 "est</w>" 已经成了一个表示"词尾 est"的子词单元，能复用在所有以 est 结尾的词。

**第 5 步：**
- `l o` 出现在 low(5) + lower(2) = 7 次 → 合并成 `lo`。

继续下去，`low`、`new` 等高频片段会逐渐长出来。**关键观察**：

1. **高频整词**（如 low）几步之内就被合并成单个 token；
2. **共享词缀**（如 est、er）被抽取成可复用单元；
3. **从没见过的新词**，比如测试时来了个 "lowest"，BPE 会用已学的规则把它切成 `low` + `est</w>`——**即使训练时没出现过 "lowest" 这个整词，也不会变成 `[UNK]`**。这就是 BPE 解决 OOV 的机制：任何词最差也能退回到字符级别拼出来。

**编码（推理时）**：拿到新文本，先拆成字符，然后**按训练时学到的合并规则的顺序**依次应用合并，能合的就合，最终得到 token 序列，再查表换成整数 ID。

### 3. 来源与历史动机
- 原始压缩算法论文：Philip Gage, *A New Algorithm for Data Compression*, 1994（C Users Journal）。
- 引入 NLP 的关键论文：Sennrich, Haddow, Birch, *Neural Machine Translation of Rare Words with Subword Units*, ACL 2016。
- 链接：https://arxiv.org/abs/1508.07909
- 后续变体：Google 的 **WordPiece**（BERT 用，按似然增益而非纯频率合并）、**Unigram LM / SentencePiece**（Kudo 2018，反向地从大词表里删词）、GPT-2/GPT-3/GPT-4 用的 **byte-level BPE**（在字节而非字符上做 BPE，彻底消灭 OOV，任何 Unicode 都能编码）。

历史背景：2016 年神经机器翻译正崛起，但 OOV 是老大难——人名、地名、复合词、罕见形态词一律 `[UNK]`，翻译质量被严重拖累。Sennrich 把 1994 年的老压缩算法 BPE 搬进 NMT，用一个简单贪心策略同时解决了词表大小和 OOV 两个问题，立刻成为标配。今天几乎所有大模型的第一道工序都是某种子词切分。

### 4. 核心用途
**用途一：控制词表大小与计算成本。** 现代 LLM 词表通常 3 万～10 万（GPT-4 的 cl100k 约 10 万）。词表大小直接决定 embedding 矩阵和输出投影层的参数量，以及最后 Softmax 的计算量。子词让我们在"几万"这个量级就覆盖几乎所有文本。

**用途二：优雅处理任意新词、拼写错误、多语言。** byte-level BPE 能编码任何 Unicode 字符串——中文、emoji、代码、URL、乱码，统统不会 `[UNK]`。这是大模型能"什么都聊"的底层前提。

**用途三：决定"上下文窗口"和"按 token 计费"的真实含义。** 你听到的"GPT 支持 128k 上下文"，单位是 token 不是字。API 按 token 计费，理解 tokenization 才知道为什么一段中文和等长英文消耗的 token 数差很多——很多中文字会被切成多个字节级 token。

### 5. 反直觉点与常见误区

**反直觉点：token 不等于"词"，甚至不等于"一个完整的字"。** 一个英文常见词可能是 1 个 token，而一个生僻汉字可能被 byte-level BPE 切成 2～3 个 token（因为一个汉字 UTF-8 编码是 3 字节）。所以"模型读了多少字"和"花了多少 token"经常对不上。著名的例子：" SolidGoldMagikarp" 这类训练数据里出现过但极罕见的字符串会变成一个孤立 token，触发模型的诡异行为（glitch token）。

**常见误区：**
- **误区一：「BPE 是按语义/语法切词的」。** 错。BPE 纯粹按**统计频率**贪心合并，切出来的子词边界经常不符合语言学词素。"东京" 可能被切得乱七八糟，它不懂语义，只懂"谁和谁常一起出现"。
- **误区二：「token 越少越好，所以应该尽量按词切」。** 不对。词表大小和序列长度是一对需要权衡的量。词表太大→参数和 Softmax 爆炸 + OOV；序列太长→注意力 $O(n^2)$ 爆炸。子词是这条权衡曲线上的一个工程甜点，不是越靠某一端越好。

### 6. 小实验
不用代码，纯手算体会"合并规则可复用到新词"：

假设你已经学到这三条有序合并规则：
1. `e` + `s` → `es`
2. `es` + `t` → `est`
3. `l` + `o` → `lo`

现在来一个训练时**没出现过**的新词 "lost"，拆成字符：`l o s t`。
- 应用规则 1（es）：没有相邻的 `e s`，跳过。
- 应用规则 2（est）：没有 `es t`，跳过。
- 应用规则 3（lo）：发现 `l o` 相邻 → 合并 → `lo s t`。

最终 "lost" → `lo`, `s`, `t` 三个 token。**没有 `[UNK]`！** 一个没见过的词被已有规则自动降级成了认识的碎片。你可以再随便编个词（比如 "slow"）走一遍流程，感受规则的"组合泛化"能力。

如果想用真代码，三行即可：`pip install tiktoken`，然后 `import tiktoken; enc = tiktoken.get_encoding("cl100k_base"); print(enc.encode("我爱Transformer"))`，亲眼看看一句中英混合被切成几个 ID。

### 7. 批判性思考

**局限性：**
- BPE 的切分对**数字和数学**很不友好——"123456" 可能被切成奇怪的几段，这被认为是 LLM 算术能力差的部分原因。
- 切分依赖训练语料分布。语料里某语言占比低，它的字就被切得更碎、更"贵"，造成**语言间的不公平**（小语种用户花更多 token、更早撞上下文上限）。
- 切分是**固定的、预处理阶段**决定的，模型本身无法端到端学习"怎么切最好"。这是个被诟病的"非神经"环节。

**深度问题：**
既然 tokenization 这么多毛病，能不能干脆**不要它**，让模型直接吃原始字节（byte-level / token-free，如 ByT5、MEGABYTE、以及近期的 BLT 字节潜变量 Transformer）？这样彻底消除 OOV 和语言不公，但序列会变得极长，又把压力还给了 $O(n^2)$ 注意力。这正是为什么"高效注意力"（类目 19）和"tokenization"是一对此消彼长的研究方向——你怎么权衡？

### 8. 和后续知识的连接
- **紧接着的下一步是「词嵌入 / Embedding 层」**：tokenization 给出的是整数 ID，Embedding 层再用一张可学习的查找表把每个 ID 映射成稠密向量（用到我们学过的「向量」「矩阵」）。Tokenization 是"切 + 编号"，Embedding 是"编号 → 向量"，两者串联起来才完成"文字 → 数字"的全过程。
- **位置编码**（类目 14）作用在 token 序列上——先有 token，才谈得上"第几个 token"。
- **注意力复杂度**（类目 15）：序列长度 $n$ 就是 token 数，tokenization 的好坏直接决定 $n$ 的大小，从而影响 $O(n^2)$ 成本。
- **自回归生成 / 采样策略**（类目 18）：模型每一步预测的是"下一个 token 的概率分布"，Softmax 的维度正是词表大小。生成结束后还要做反向的 detokenize 把 ID 串拼回人类可读文本。
