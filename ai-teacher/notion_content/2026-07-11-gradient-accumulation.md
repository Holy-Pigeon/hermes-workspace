## 今日知识点：梯度累积（Gradient Accumulation）——显存不够，就把大 batch 拆成几口慢慢咽

### 所属类目
类目 8：优化算法

### 问题入口
训练大模型时，你几乎总会撞上同一堵墙：**显存不够**。

论文里说这个模型要用 batch size = 512 才能训得稳，可你手上的显卡塞进 32 个样本就 OOM（Out of Memory，显存溢出）了。难道 batch size 只能被显卡容量卡死？大 batch 带来的稳定梯度、平滑训练曲线，就只能眼睁睁放弃？

这个矛盾非常具体：**我们想要的「有效 batch」远大于显卡一次能装下的「物理 batch」**。梯度累积就是专门解决这堵墙的技巧。

### 朴素想法
最直接的想法有两个：

1. **就用小 batch 训练。** 显卡只装得下 32，那就 batch size = 32，认了。
2. **换更大的显卡 / 加更多卡。** 钱能解决的问题就不是问题。

这两个想法都不是「错」，但都有代价——第一个牺牲了训练质量，第二个牺牲了钱包。有没有办法用手上的小显卡，训出大 batch 的效果？

### 卡住的地方
先看朴素想法 1 为什么不够。小 batch 的梯度是「用少量样本估计出来的方向」，噪声大。回忆我们学过的随机梯度下降（SGD）：每一步的梯度是真实梯度的一个**带噪声的估计**，batch 越小噪声越大。噪声大意味着：

- 训练曲线抖动剧烈，loss 忽上忽下；
- 对学习率极其敏感，稍微调大就发散；
- 某些依赖批统计量的技术（如 Batch Normalization）在极小 batch 下会失真。

对很多大模型（尤其是 Transformer），论文报告的超参数是在大 batch 下调出来的。你换成 batch = 32，很可能复现不出效果。

而朴素想法 2（加卡）不是每个人都负担得起。于是问题变成：**能不能不增加显存，就等价地放大 batch？**

关键洞察在这里：一次性算 512 个样本的梯度，和分 16 次、每次算 32 个样本的梯度再加起来，**在数学上是同一件事**——只要 loss 是对样本求和/平均的。梯度是线性算子，可以拆开累加。这就是梯度累积的全部秘密。

### 1. 概念
**梯度累积**：把一个「大 batch」拆成若干个「小 micro-batch」，逐个做前向和反向传播，但**不立刻更新参数**，而是把每个 micro-batch 算出的梯度**累加**到一起；等累积够了预定的次数，再用累加后的总梯度做一次参数更新。

通俗理解：假设你要搬 512 块砖到楼上，但你一次只扛得动 32 块。你不会因此只搬 32 块就收工，而是分 16 趟搬，每趟 32 块，搬完 16 趟才算「完成一个大批次」。梯度累积就是把「一次更新」拆成 16 趟「攒梯度」，攒够了再走一步。

但类比要回到数学。设损失函数对一个 batch 是各样本损失的平均。对一个大 batch $B$（大小 $N$）的梯度：

$$
g_B = \frac{1}{N} \sum_{i=1}^{N} \nabla_\theta \ell(x_i, \theta)
$$

其中 $\ell(x_i, \theta)$ 是第 $i$ 个样本的损失，$\theta$ 是模型参数，$\nabla_\theta$ 是对参数求梯度。

现在把这 $N$ 个样本切成 $K$ 个 micro-batch，每个大小 $m = N/K$。第 $k$ 个 micro-batch 的梯度是：

$$
g_k = \frac{1}{m} \sum_{i \in B_k} \nabla_\theta \ell(x_i, \theta)
$$

那么大 batch 的梯度可以由 micro-batch 梯度平均得到：

$$
g_B = \frac{1}{K} \sum_{k=1}^{K} g_k
$$

**这个等式是梯度累积成立的全部数学基础**：大 batch 的平均梯度，等于 $K$ 个 micro-batch 平均梯度的平均。只要我们把每个 micro-batch 的梯度加起来再除以 $K$，就得到了和一次性算大 batch **完全相同**的梯度——却只需要一个 micro-batch 的显存。

### 2. 推导与证明
我们证明「分 $K$ 次累加」和「一次性算大 batch」的梯度严格相等。

关键步骤：

1. **写出大 batch 的平均损失。** 大 batch 损失 $L_B = \frac{1}{N}\sum_{i=1}^{N}\ell(x_i,\theta)$。因为求梯度是线性运算，$\nabla_\theta L_B = \frac{1}{N}\sum_{i=1}^{N}\nabla_\theta \ell(x_i,\theta)$。这一步用到「和的梯度等于梯度的和」——这正是我们学过的梯度的线性性质。

2. **把求和按 micro-batch 分组。** 把 $\{1,\dots,N\}$ 分成 $K$ 组，每组 $m=N/K$ 个。求和可以先组内加、再组间加：
$$
\sum_{i=1}^{N}\nabla_\theta \ell(x_i,\theta) = \sum_{k=1}^{K}\sum_{i \in B_k}\nabla_\theta \ell(x_i,\theta)
$$
为什么能这么做？因为加法满足结合律，分组求和不改变总和。

3. **凑出每个 micro-batch 的平均梯度。** 第 $k$ 组内部的和是 $m \cdot g_k$（因为 $g_k=\frac{1}{m}\sum_{i\in B_k}\nabla_\theta\ell$）。代回去：
$$
\nabla_\theta L_B = \frac{1}{N}\sum_{k=1}^{K} m\, g_k = \frac{m}{N}\sum_{k=1}^{K} g_k = \frac{1}{K}\sum_{k=1}^{K} g_k
$$
最后一步用了 $m/N = 1/K$。证毕——**累加 $K$ 个 micro-batch 梯度再除以 $K$，就是大 batch 梯度本身，一点近似都没有。**

工程实现上的循环长这样（伪代码，说明机制）：

```python
optimizer.zero_grad()                      # 清空累积的梯度
for k in range(K):                         # 遍历 K 个 micro-batch
    x, y = get_micro_batch(k)
    loss = criterion(model(x), y) / K      # 关键：loss 除以 K
    loss.backward()                        # 梯度累加进 .grad，不清零
optimizer.step()                           # K 次攒完，才更新一次
optimizer.zero_grad()                      # 为下一个大 batch 清零
```

这里有个**极易踩坑的细节**：`loss.backward()` 在 PyTorch 里默认是把新梯度**加到**已有的 `.grad` 上（而不是覆盖），所以我们「什么都不做」就自动累积了。但正因为它是「加」，每个 micro-batch 的 loss 必须先除以 $K$，否则累加出来的是 $K$ 倍的梯度，等价于把学习率放大了 $K$ 倍，很容易训飞。

### 3. 来源与历史动机
- 论文：梯度累积没有单独的「开山论文」，它是随分布式/大批量训练自然出现的工程技巧。相关的理论背景可追溯到大批量训练研究，代表性文献是 Goyal 等人的《Accurate, Large Minibatch SGD: Training ImageNet in 1 Hour》。
- 作者：Priya Goyal, Piotr Dollár, Ross Girshick 等（Facebook AI Research）。
- 时间：2017 年。
- 链接：https://arxiv.org/abs/1706.02677

历史背景：2017 年前后，深度学习模型和数据集迅速变大，研究者发现「大 batch + 合适学习率」能显著加速并稳定训练。但显卡显存增长远跟不上模型胃口，尤其是 Transformer 崛起后，注意力的显存开销随序列长度平方增长。梯度累积让「显存受限的普通研究者」也能复现「工业界大 batch」的结果，成了几乎每个训练框架（HuggingFace Trainer、DeepSpeed、Megatron）的标配开关（通常叫 `gradient_accumulation_steps`）。

### 4. 核心用途
用途一：**在小显卡上复现大 batch 训练。** 这是最主要的场景。想要有效 batch = 512，显卡只装得下 32，就设 `gradient_accumulation_steps = 16`。论文超参数原样照抄即可复现。

用途二：**训练超大模型 / 超长序列。** 大语言模型微调时，单条样本就可能占满显存，物理 batch 只能是 1。这时梯度累积几乎是唯一能放大有效 batch 的手段，配合梯度检查点（gradient checkpointing）一起用。

用途三：**在多卡分布式训练里进一步放大有效 batch。** 数据并行时每张卡各算一部分，再在卡间同步梯度；梯度累积正交叠加在上面：有效 batch = 单卡物理 batch × 卡数 × 累积步数。三个乘子任你调配。

### 5. 反直觉点与常见误区
反直觉点：**梯度累积「不省」计算量，只省显存。** 很多人以为它是加速技巧——恰恰相反，它通常还**略微变慢**。因为你把一次大矩阵运算拆成了 $K$ 次小运算，显卡的并行度利用不足，吞吐反而下降。你用「时间」换了「显存」。它的价值是「让原本训不了的能训了」，而不是「让训得更快」。

常见误区：
- **误区一：忘记把 loss 除以 $K$。** 前面强调过：不除以 $K$，累加梯度会是 $K$ 倍，等于偷偷放大学习率，训练极易发散。或者等价地在 `optimizer.step()` 前对梯度做平均。
- **误区二：以为梯度累积和大 batch 完全等价。** 在纯 SGD 下，梯度部分严格等价（我们证过）。但一旦模型里有 **Batch Normalization**，就**不等价**了——BN 的均值方差是在每个 micro-batch 内部统计的，micro-batch 只有 32 个样本，统计量和真正 512 的 batch 不同。所以带 BN 的网络，梯度累积无法完全复现大 batch。（Transformer 用 LayerNorm/RMSNorm，逐样本归一化，不受这个问题影响——这也是它比 BN 更适合累积的原因之一。）
- **误区三：把累积步数当作免费午餐无限调大。** 步数越大，参数更新越稀疏（每 $K$ 个 micro-batch 才走一步），一个 epoch 的实际更新次数减少，收敛到目标可能需要更多 epoch。

### 6. 小实验
不用 GPU，用心算就能验证「累加 = 大 batch」这个核心等式。

设想一个只有一个参数 $\theta$ 的模型，4 个样本的单样本梯度分别是：$2, 4, 6, 8$。

- **一次性大 batch（$N=4$）**：平均梯度 $=(2+4+6+8)/4 = 5$。
- **梯度累积（$K=2$，每个 micro-batch 2 个样本）**：
  - micro-batch 1：$(2+4)/2 = 3$
  - micro-batch 2：$(6+8)/2 = 7$
  - 两者平均：$(3+7)/2 = 5$。

两种算法都得到 5，**分毫不差**。再故意犯个错验证误区一：如果 micro-batch 的 loss 忘了除以 $K=2$，你累加的是 $6 + 14 = 20$，是正确梯度 5 的 4 倍（$=2K \cdot 5$，因为既没在 micro-batch 内平均、又漏了除 $K$）——学习率被悄悄放大，直观感受到「忘除 K」有多危险。

### 7. 批判性思考
局限性：
- **不省时间，甚至更慢**：拆小运算牺牲了显卡并行度，吞吐下降。
- **和 BN 不完全兼容**：如上所述，带 Batch Normalization 的网络无法用累积严格复现大 batch。
- **更新变稀疏**：同样的 wall-clock 时间里参数更新步数变少，可能需要更多训练时间才收敛。
- **超大 batch 本身有收益递减**：即使用累积做到了 batch = 4096，研究（如 OpenAI 的 gradient noise scale 工作）表明超过某个「临界 batch size」后，增大 batch 的边际收益迅速衰减——不是越大越好。

深度问题：如果梯度累积能无损放大 batch，为什么工业界还要花大价钱堆几千张 GPU 做真正的大 batch 数据并行，而不是用少量卡 + 巨大累积步数？（提示：想想 wall-clock 训练时间、每步更新的墙钟间隔、以及一个模型要在合理时间内训完需要多少「真实并行算力」——累积换来了显存，但换不来时间。）

### 8. 和后续知识的连接
- **上承随机梯度下降（SGD）与 mini-batch**：梯度累积本质是「把 mini-batch 拆得更细再拼回来」，理解 SGD 的梯度是「带噪声的估计」是理解它为什么有用的前提。
- **上承梯度的线性性质**：整个技巧成立完全依赖「和的梯度 = 梯度的和」，这是我们在链式法则、反向传播里反复用到的基本性质。
- **和混合精度训练、梯度检查点是「三件套」**：现代大模型训练里，梯度累积（省激活/batch 显存）、混合精度（省数值位宽）、梯度检查点（用重算省激活显存）经常同时开启，共同压低单卡显存占用。
- **和分布式数据并行正交叠加**：真正训练大语言模型时，有效 batch = 卡数 × 单卡物理 batch × 累积步数，三个维度共同决定最终的有效 batch，进而配合缩放定律（Scaling Laws）去规划算力预算。
- **和 LayerNorm/RMSNorm 的适配性**：因为 Transformer 用逐样本归一化而非 BN，梯度累积对它几乎无损——这也是为什么大模型训练能如此依赖累积。
