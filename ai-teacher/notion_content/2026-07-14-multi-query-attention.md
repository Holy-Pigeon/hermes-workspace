## 今日知识点：多查询注意力（Multi-Query Attention, MQA）——所有头共用同一套钥匙和仓库

### 所属类目
类目 19：高效注意力

### 问题入口
我们之前学过 KV Cache：自回归生成时，为了不让模型每吐一个字都把前文重算一遍，我们把每个已生成 token 的 Key 和 Value 缓存下来。这个技巧让推理快了很多，但它带来一个新的、越来越致命的瓶颈——**显存**。

设想一个真实场景：一个 32 层、每层 32 个注意力头、隐藏维度 $d=4096$ 的模型，要生成一段 8192 token 的长文本，batch size 为 16。请先别急着往下看，估一下：光是 KV Cache 要占多少显存？

答案会吓你一跳：几十 GB，往往比模型权重本身还大。更糟的是，生成阶段每吐一个 token，都要把整个 KV Cache 从显存搬进计算单元读一遍——**推理速度不再受限于算力，而是受限于「把 KV 搬来搬去」的内存带宽**。

问题变成了：**能不能在几乎不损失质量的前提下，把 KV Cache 砍掉一大截？** 这就是 MQA 要解决的。

### 朴素想法
既然 KV Cache 太大，最直接的想法有几个：

1. **缓存少存一点**：只缓存最近若干个 token 的 KV，远处的丢掉。——这其实就是我们学过的 Sliding Window Attention 的思路，但它会真的丢失长程信息。
2. **把 KV 压缩存储**：用量化（我们学过 Quantization）把 KV 从 16 位压到 8 位甚至 4 位。——有效，但压缩率有限，且有精度损失。
3. **少算几个头**：干脆把注意力头的数量从 32 砍到 8。——但头数减少会明显削弱模型从多个子空间看问题的能力（回忆「多子空间表示」那一课）。

这些想法都在「减小 KV 的绝对大小」上打转。但有没有一种办法，**保留全部 32 个头的表达力，同时又让 KV 缓存缩小到原来的 1/32**？

### 卡住的地方
朴素想法的共同问题是：它们要么损失信息（少存、少算），要么压缩率有限（量化）。它们都默认了一件事——**每个注意力头都必须有自己独立的 Key 和 Value**。

让我们回到多头注意力的结构看看这个假设。标准多头注意力（MHA）里，对第 $h$ 个头：

$$
Q_h = X W_h^Q, \quad K_h = X W_h^K, \quad V_h = X W_h^V
$$

这里 $X$ 是输入（形状 $[n, d]$，$n$ 是序列长度），$W_h^Q, W_h^K, W_h^V$ 是第 $h$ 个头各自独立的投影矩阵。有 $H$ 个头，就有 $H$ 套不同的 $K$ 和 $V$。KV Cache 要缓存的正是所有头、所有位置的 $K_h$ 和 $V_h$，其大小正比于 $H$（头数）。

关键的追问来了：**这 $H$ 套各不相同的 Key 和 Value，真的每一套都不可或缺吗？还是说，头与头之间的差异，主要体现在 Query 上，而 Key/Value 其实高度冗余？**

MQA 大胆地押注了后者。

### 1. 概念
**多查询注意力（Multi-Query Attention, MQA）**：让所有注意力头**共享同一套 Key 和 Value**，但**每个头保留各自独立的 Query**。

也就是说，原来是 $H$ 个 $Q$、$H$ 个 $K$、$H$ 个 $V$；MQA 变成 $H$ 个 $Q$、但只有 $1$ 个 $K$、$1$ 个 $V$。

正式表达。对第 $h$ 个头：

$$
Q_h = X W_h^Q, \quad K = X W^K, \quad V = X W^V
$$

注意 $K$ 和 $V$ 的投影矩阵 $W^K, W^V$ **不再带下标 $h$**——它们是全体头共用的。每个头的注意力照常计算，只是大家去查的是同一个「钥匙库」和「仓库」：

$$
\text{head}_h = \text{softmax}\!\left(\frac{Q_h K^\top}{\sqrt{d_k}}\right) V
$$

其中 $d_k$ 是每个头的维度，$\sqrt{d_k}$ 是我们学过的缩放因子（防止点积过大把 softmax 推到饱和区）。

**通俗理解**：想象一个图书馆里有 32 位读者（32 个头）。标准 MHA 相当于给每位读者配一整套私人的书目卡片（Key）和一整座私人书库（Value）——32 套，占地极大。MQA 说：书库和卡片目录**全馆共用一套**就够了，每位读者只保留自己**独特的检索问题**（Query）。大家带着各自不同的问题，去查同一座公共图书馆。

但类比到此为止，回到机制：为什么这样做质量掉得很少？因为注意力的「看哪里」这个决策，主要由 $Q$ 和 $K$ 的匹配决定，而**头之间的多样性，很大程度上已经由各自独立的 $Q_h$ 提供了**。共享 $K, V$ 损失的是「不同头用不同方式编码被查内容」的能力，实践证明这部分冗余度很高。

### 2. 推导与证明
我们来定量算一下 MQA 到底省了多少，这才是它的价值所在。

关键步骤：

1. **算标准 MHA 的 KV Cache 大小。** 缓存需要存所有层、所有头、所有位置的 K 和 V。单个样本的 KV Cache 元素个数为：
$$
\text{Size}_{MHA} = 2 \cdot L \cdot H \cdot n \cdot d_k
$$
其中 $L$ 是层数，$H$ 是头数，$n$ 是序列长度，$d_k$ 是每头维度，因子 $2$ 是因为要同时存 K 和 V。注意 $H \cdot d_k = d$（模型隐藏维度），所以也可写成 $2 L n d$。
   - **为什么这么算**：每个位置每个头都要独立缓存一份 K 和一份 V，缺一不可，所以维度全乘起来。

2. **算 MQA 的 KV Cache 大小。** 因为所有头共享一套 K/V，头数 $H$ 从公式里消失了：
$$
\text{Size}_{MQA} = 2 \cdot L \cdot 1 \cdot n \cdot d_k = 2 L n d_k
$$
   - **为什么这么做**：既然全体头只用一份 K、一份 V，缓存自然就只存一份。

3. **算压缩比。** 两者相除：
$$
\frac{\text{Size}_{MHA}}{\text{Size}_{MQA}} = \frac{2 L H n d_k}{2 L n d_k} = H
$$
   - **结论**：KV Cache 精确缩小为原来的 $1/H$。头数是 32，缓存就砍到 1/32。这不是近似，是结构性的精确结果。

4. **它如何转化为速度？** 生成阶段是**内存带宽受限（memory-bound）**的：瓶颈不是算得慢，而是把 KV 从显存读进来慢。KV Cache 小 $H$ 倍，意味着每步要搬运的数据少 $H$ 倍，于是解码吞吐大幅提升（实际论文报告约 10 倍量级的解码加速，具体依硬件而定）。
   - **为什么带宽是瓶颈**：自回归生成时 batch 内每个序列一次只处理 1 个新 token，计算量很小，但要读取的历史 KV 很大——算力闲着，带宽跑满。缩小 KV 直接命中这个瓶颈。

5. **参数量的副产品。** $W^K, W^V$ 从 $H$ 套变 1 套，注意力层的参数也略有减少（但这是次要收益，主收益是推理时的 KV Cache 与带宽）。

一个反向验证：如果共享 K/V 真的毫无代价，那 MHA 为什么当初要设计成每头独立？因为训练时 MHA 表达力确实更强、质量上限更高。MQA 是在「质量」和「推理成本」之间做了一个**明显偏向成本**的权衡——这也正是后来 GQA 出现的原因。

### 3. 来源与历史动机
- 论文：《Fast Transformer Decoding: One Write-Head is All You Need》
- 作者：Noam Shazeer（Transformer、MoE 等多项工作的核心作者）
- 时间：2019 年
- 链接：https://arxiv.org/abs/1911.02150

历史背景：2019 年，Transformer 已经统治了 NLP，但把它部署成**实时生成服务**时，工程师们撞上了一堵墙——解码速度太慢，而慢的根源不是浮点运算，是 KV Cache 的内存带宽。Shazeer 在这篇短小精悍的论文里指出：注意力有多个「读头」（Query）是必要的，但「写头」（产生 K/V 的部分）其实一个就够（论文标题「One Write-Head is All You Need」正是此意，戏仿了「Attention is All You Need」）。这个洞察让 MQA 成为后来一系列大模型（如 PaLM、Falcon）推理优化的基石，也直接催生了 2023 年的 GQA——我们上一课已经学过的那个「折中方案」。

### 4. 核心用途
用途一：**长上下文推理的显存救星。** 当模型要处理几万甚至几十万 token 的上下文时，MHA 的 KV Cache 会撑爆显存。MQA 把它砍到 1/H，让同样的硬件能吃下长得多的上下文，或在相同上下文下服务更大的 batch。

用途二：**高吞吐在线服务。** 面向海量用户的聊天/补全服务，追求的是每秒处理的 token 数。MQA 缩小了每步搬运的数据量，直接提升解码吞吐、降低单位推理成本——这是真金白银的部署优化。

用途三：**为 GQA 提供了理论端点。** MQA（所有头共享 1 组 KV）和 MHA（每头一组 KV）是两个极端；GQA 通过设组数介于两者之间，在质量和成本上取中。理解 MQA 是理解现代主流大模型（如 Llama 2/3、Mistral 都用 GQA）注意力设计的必经一步。

### 5. 反直觉点与常见误区
反直觉点：**减少 KV 头几乎不加速训练，主要加速的是推理。** 很多人以为「参数少了、KV 少了，训练也会快很多」。但训练是**计算受限（compute-bound）**的，一次并行处理整个序列，算力跑满，KV Cache 甚至不需要（训练时不做增量缓存）。MQA 省的是**推理时**的内存带宽。它是一个「推理优化」而非「训练优化」。这个区分极其重要。

常见误区：
- **误区一：「MQA 减少了 Query 头，所以头变少了」。** 恰恰相反——MQA 保留全部 $H$ 个 Query 头，减的是 Key/Value 头（从 $H$ 减到 $1$）。模型「看哪里」的多样性由 Query 提供，这部分一点没动。
- **误区二：「MQA 一定会明显掉点」。** 早期确实观察到 MQA 相比 MHA 有小幅质量下降和训练不稳定，这正是 GQA 折中的动机。但配合适当的训练技巧，MQA 的质量损失往往远小于它带来的成本收益，很多任务上几乎无感。是否值得，取决于你更在意质量上限还是部署成本。

### 6. 小实验
不用 GPU，用心算 + 一段极短代码就能感受 MQA 的威力。

先心算：一个模型 $L=32$ 层，$H=32$ 头，每头维度 $d_k=128$（即 $d=4096$），要缓存 $n=8192$ 个 token，batch size $b=16$，用 16 位（2 字节）存储。

MHA 的 KV Cache 字节数 $= 2 \times L \times H \times d_k \times n \times b \times 2$ 字节。代入：$2 \times 32 \times 32 \times 128 \times 8192 \times 16 \times 2 \approx 5.5 \times 10^{11}$ 字节 $\approx 512$ GB。这远超单卡显存！

MQA 把 $H=32$ 换成 $1$，直接除以 32，变成约 **16 GB**——瞬间可行。

代码验证（几行 numpy 就能跑）：

```python
import numpy as np

L, H, dk, n, b, bytes_per = 32, 32, 128, 8192, 16, 2

def kv_cache_bytes(kv_heads):
    # 2 = K 和 V 两份
    return 2 * L * kv_heads * dk * n * b * bytes_per

mha = kv_cache_bytes(H)   # 每头一套 KV
mqa = kv_cache_bytes(1)   # 所有头共享一套 KV
gqa = kv_cache_bytes(8)   # GQA：8 组，作为对照

print(f"MHA KV Cache: {mha/1e9:.1f} GB")
print(f"MQA KV Cache: {mqa/1e9:.1f} GB")
print(f"GQA(8组) KV Cache: {gqa/1e9:.1f} GB")
print(f"MQA 压缩比: {mha/mqa:.0f}x")
```

跑出来 MHA 约 512 GB、MQA 约 16 GB、GQA(8) 约 128 GB，压缩比正好 32 倍。亲手看到这条 $1/H$ 的规律，比记住公式深刻得多。

### 7. 批判性思考
局限性：MQA 把权衡的旋钮拧到了「极致省成本」的一端，代价是质量与训练稳定性的一定牺牲。对质量极度敏感、又不那么在意推理成本的场景（比如离线批量推理、对准确率要求苛刻的任务），MHA 或折中的 GQA 可能更合适。换句话说，MQA 不是「更好的注意力」，而是「成本导向的注意力」——好坏取决于你的部署约束。

深度问题：既然共享 K/V 能省这么多而质量损失可控，说明标准 MHA 的 K/V 存在大量冗余。那么，**这种冗余的边界在哪里**？GQA 用「分组」在 MHA（$H$ 组）和 MQA（$1$ 组）之间连续插值，实验发现往往 4~8 组就能拿到接近 MHA 的质量和接近 MQA 的速度。这暗示了一个更深的问题：注意力机制真正需要的「独立 K/V 子空间」数量，可能远少于头数——这是否意味着，我们一直在为不必要的表达自由度付费？沿这个思路，还能不能设计出比 GQA 更聪明的 KV 共享结构（比如让 K/V 也做低秩分解，正如后来 MLA 所做的）？

### 8. 和后续知识的连接
- **向后连接**：MQA 是我们上一阶段学过的 **GQA（分组查询注意力）** 的直接前身。GQA = 介于 MHA（每头独立 KV）和 MQA（全体共享 1 组 KV）之间的连续谱。理解了 MQA 这个极端，GQA 的「折中」定位就一目了然。
- **依赖前置**：它建立在 **KV Cache**（为什么要缓存）、**多头注意力 / 多子空间表示**（头是什么、Q/K/V 的分工）、**缩放点积注意力**（每头怎么算）之上——这些我们都学过。
- **向前延伸**：这条「减少 KV 冗余」的主线还在演进。更前沿的 **MLA（Multi-head Latent Attention，DeepSeek 提出）** 把 K/V 压进一个低秩潜在空间，是 MQA 思想的进一步发展。同时，MQA/GQA 的 KV 缩减和 **FlashAttention**（省 IO）、**Quantization**（省比特）是正交的优化，实际部署中常常叠加使用。掌握 MQA，你就握住了理解「现代大模型推理为什么这么快」的一把关键钥匙。
