## 今日知识点：投机解码（Speculative Decoding）——让小模型先抢答，大模型只做批改

### 所属类目
类目 18：Decoder-only 模型

### 问题入口
上一课我们讲过 KV Cache，把"每吐一个字都重算前文"的浪费省掉了。但即便有了 KV Cache，自回归生成还有一个绕不开的硬伤：**一次前向只能吐出一个 token**。

想生成 200 个 token，就得把大模型（比如 70B 参数）从头到尾跑 200 次。每一次前向都要把几百亿参数从显存搬进计算单元。这里的瓶颈不是"算得慢"，而是"搬得慢"——大模型解码是**显存带宽受限（memory-bandwidth bound）**的：GPU 的算力大量闲置，时间几乎全花在把权重从显存读进来上。

于是有了一个看似无解的矛盾：
- 想要质量，就得用大模型；
- 大模型每步都很贵，而且必须一步一步来，无法并行。

问题来了：**能不能让大模型一次前向，验证好几个 token，从而把 200 步压缩成几十步？**

### 朴素想法
最直接的念头是："那就让大模型一次多预测几个 token 呗。" 比如让它同时输出未来 4 个位置的词。

再进一步，有人会想：既然大模型慢，**能不能干脆用一个小模型（比如 1B）来生成？** 小模型每步便宜十几倍，跑得飞快。质量差一点忍忍算了。

### 卡住的地方
两个朴素想法都撞墙：

**"一次多预测几个"不行**：自回归模型的第 $t+2$ 个 token 依赖于第 $t+1$ 个 token 的真实取值。你让它同时猜 $t+1$ 和 $t+2$，猜 $t+2$ 时它并不知道 $t+1$ 到底是什么，只能瞎蒙。这就破坏了链式依赖，质量崩塌。

**"直接用小模型"也不行**：小模型质量确实差。它在大部分"容易的位置"（比如 "the"、"is"、标点、高频搭配）表现和大模型几乎一样，但在**关键的少数难位置**上会犯错，而这些错误会顺着自回归一路放大，最终输出面目全非。

卡点的本质是：我们既要大模型的质量，又嫌它慢；小模型快，但不能信。**怎么把"快"和"可信"这两件事拆开？**

### 1. 概念

**投机解码**的核心洞察：**生成难，但验证易。**

就像做数学题——从头解一道题很费劲，但给你一份答案让你检查对错，快得多。投机解码就是把生成拆成两个角色：

- **草稿模型（draft model）**：一个小而快的模型，先"投机"地连续猜出接下来 $k$ 个 token，作为一份草稿。
- **目标模型（target model）**：那个又大又准的模型，**一次前向并行地**验证这 $k$ 个草稿 token 是否"它自己也会这么写"。

关键在于：验证这 $k$ 个 token，目标模型只需要**一次前向**。因为这 $k$ 个 token 已经被草稿模型摆在那里了，目标模型可以把它们当成"已知输入"，一次性并行算出每个位置的概率分布——这正好利用了 Transformer 训练时那种"整句并行"的能力，把原本闲置的算力用起来。

**通俗理解**：老编辑（大模型）写稿慢，就雇一个实习生（小模型）先刷刷刷写出一段草稿。老编辑不用逐字自己写，只需扫一眼草稿："对、对、对、这个词不对"——在第一个不对的地方停下、改掉，后面作废重写。只要实习生大部分时候写得还行，老编辑扫一遍的时间远小于自己逐字写的时间。

但类比到此为止，真正精妙的是**怎么"验证"才能保证最终输出和大模型单独生成的分布**一模一样——不是"差不多"，而是数学上严格等价。这靠的是下面的**投机采样（speculative sampling）**接受-拒绝规则。

正式地，设目标模型在某位置给出分布 $p(x)$，草稿模型给出 $q(x)$，草稿采样得到 token $x$：

$$
\text{接受 } x \text{ 的概率} = \min\left(1, \frac{p(x)}{q(x)}\right)
$$

若被拒绝，则从一个"修正分布"里重新采样：

$$
x' \sim \text{norm}\big(\max(0,\; p(x) - q(x))\big)
$$

其中 $\text{norm}(\cdot)$ 表示把括号里的非负函数归一化成一个概率分布（各项除以总和使其加起来为 1）。

### 2. 推导与证明

我们要证明的核心结论是：**用上面这套接受-拒绝规则采出来的 token，其分布恰好等于目标分布 $p(x)$**。这意味着投机解码不是近似——它输出的每个 token，在统计上和"用大模型老老实实一步步采样"完全不可区分。

关键步骤：

1. **草稿阶段**：草稿模型从 $q$ 中采出 $x$。此时 $x$ 服从 $q$。
2. **接受判定**：以概率 $\min(1, p(x)/q(x))$ 接受。
3. **计算某个具体 token $x$ 最终被输出的总概率**，它由"接受"和"拒绝后恰好重采到 $x$"两部分组成。

先看被接受的那部分。$x$ 先被 $q$ 采中（概率 $q(x)$），再被接受（概率 $\min(1, p(x)/q(x))$）：

$$
P_{\text{accept}}(x) = q(x)\cdot \min\left(1, \frac{p(x)}{q(x)}\right) = \min\big(q(x),\, p(x)\big)
$$

再看被拒绝后重采的那部分。任意一次草稿被拒绝的总概率是：

$$
P_{\text{reject}} = \sum_{y} q(y)\left(1 - \min\left(1, \frac{p(y)}{q(y)}\right)\right) = \sum_y \max\big(0,\, q(y)-p(y)\big)
$$

拒绝后从修正分布 $\text{norm}(\max(0, p-q))$ 重采到 $x$ 的概率是 $\dfrac{\max(0,\, p(x)-q(x))}{\sum_y \max(0,\, p(y)-q(y))}$。注意到分母恰好等于上面的 $P_{\text{reject}}$（因为 $\max(0,q-p)$ 与 $\max(0,p-q)$ 对所有 $y$ 求和相等——两者都等于 $p,q$ 之间"错开"的总质量），所以拒绝这条路径贡献的概率是：

$$
P_{\text{resample}}(x) = P_{\text{reject}} \cdot \frac{\max(0,\, p(x)-q(x))}{P_{\text{reject}}} = \max\big(0,\, p(x)-q(x)\big)
$$

4. **两部分相加**：

$$
P(x) = \min\big(q(x), p(x)\big) + \max\big(0,\, p(x)-q(x)\big)
$$

分两种情况看这个式子：
- 若 $p(x) \geq q(x)$：$\min = q(x)$，$\max = p(x)-q(x)$，加起来 $= p(x)$。
- 若 $p(x) < q(x)$：$\min = p(x)$，$\max = 0$，加起来 $= p(x)$。

两种情况都得到 $P(x) = p(x)$。**证毕**——最终分布严格等于目标分布。

这就是投机解码"免费加速"的数学根基：**在不改变输出分布的前提下拿到加速**。一次目标模型前向若能接受 $k$ 个草稿 token 中的前 $m$ 个（再加验证时"顺手"生成的 1 个），这一步就相当于走完了 $m+1$ 步，把总步数压缩下来。

### 3. 来源与历史动机
- 论文一：《Fast Inference from Transformers via Speculative Decoding》
- 作者：Yaniv Leviathan, Matan Kalman, Yossi Matias（Google Research）
- 时间：2022 年 11 月（ICML 2023）
- 链接：https://arxiv.org/abs/2211.17192

- 论文二（几乎同期独立提出）：《Accelerating Large Language Model Decoding with Speculative Sampling》
- 作者：Charlie Chen 等（DeepMind）
- 时间：2023 年 2 月
- 链接：https://arxiv.org/abs/2302.01318

历史背景：2022 年前后，GPT-3 级别的大模型开始大规模部署，推理延迟和成本成了落地的核心痛点。人们发现大模型解码是带宽受限的——算力大量闲置。Leviathan 等人意识到，可以把这些闲置算力用来"并行验证"，而不是"并行生成"，并用接受-拒绝采样保证输出分布不变。这一思想优雅之处在于：它不牺牲任何质量，是真正的"白捡加速"，因此被 vLLM、TensorRT-LLM 等主流推理框架迅速采纳。

### 4. 核心用途

用途一：**大模型在线推理加速**。ChatGPT 这类交互式服务对首字延迟和吐字速度敏感。用同系列小模型当草稿，实测常见 2–3 倍吞吐提升，且输出与原模型不可区分。

用途二：**降低推理成本**。步数减少意味着大模型前向次数减少，GPU 占用时间缩短，单位 token 成本下降。对按调用量计费的 API 服务，这是直接的利润。

用途三：**衍生出无需独立草稿模型的变体**。Medusa（给目标模型加几个并行预测头当草稿）、自投机（self-speculative，用模型自己的浅层当草稿）、EAGLE 等，都是"生成难、验证易"这一思想的延伸，避免了额外维护一个草稿模型的成本。

### 5. 反直觉点与常见误区

反直觉点：**投机解码不减少总计算量，反而增加了计算量（草稿模型的前向 + 目标模型验证被拒绝的那些 token 白算了），却依然更快。** 因为大模型解码卡在显存带宽而非算力上——把闲置的算力拿来并行验证，等于用"免费的算力"换"宝贵的时间"。这也解释了为什么它在带宽受限的大模型上收益大，而在算力受限的小模型上可能反而变慢。

常见误区：
- **误区一：以为投机解码是近似、会掉质量。** 恰恰相反，标准投机采样在数学上严格保证输出分布等于目标模型，质量零损失（浮点误差除外）。
- **误区二：以为草稿模型越强越好。** 草稿模型太强（太接近目标模型）本身就慢，失去了"快"的意义；太弱则接受率低、频繁被拒、白算太多。存在一个甜点区，通常草稿模型比目标模型小 10–20 倍最划算。
- **误区三：以为加速倍数是固定的。** 加速比强烈依赖"接受率"，而接受率取决于任务。代码、格式化文本这类高度可预测的内容接受率高、加速猛；开放创作、高温度采样时草稿常被拒，加速有限。

### 6. 小实验

不用 GPU，用两个骰子手动体会接受-拒绝规则如何"纠偏"。

设某位置只有两个候选词 A、B。目标模型（大）的真实分布 $p$：$p(A)=0.6, p(B)=0.4$。草稿模型（小）的分布 $q$：$q(A)=0.8, q(B)=0.2$——草稿明显偏爱 A。

现在跑规则：
1. 草稿采样。80% 出 A，20% 出 B。
2. 若草稿出的是 A：接受概率 $\min(1, p(A)/q(A)) = \min(1, 0.6/0.8) = 0.75$。即 A 有 25% 概率被拒。
3. 若草稿出的是 B：接受概率 $\min(1, p(B)/q(B)) = \min(1, 0.4/0.2) = 1$。B 永远被接受。

手算最终输出 A 的概率：$P(A) = \min(0.8, 0.6) + \max(0, 0.6-0.8) = 0.6 + 0 = 0.6$。正好等于 $p(A)=0.6$！

直觉解读：草稿高估了 A，接受规则就"抽掉"一部分 A（那 25% 的拒绝）；拒绝后从修正分布 $\text{norm}(\max(0, p-q))$ 里重采——此处 $p(A)-q(A)<0$ 归零、$p(B)-q(B)=0.2>0$，所以拒绝后必定输出 B，把被抽掉的概率精准补给了 B。系统自动把草稿的偏差纠回了真实分布。

### 7. 批判性思考

局限性：
- **贪心解码时收益打折**：温度为 0（总取最高概率）时，只有草稿和目标"恰好想法一致"才接受，长文本里一处分歧就截断，接受率受限。
- **需要两个模型共存**：草稿模型也要占显存、也要维护 KV Cache，小 batch 下划算，但在超大 batch（本已把算力吃满、变回算力受限）场景收益会缩水甚至转负。
- **接受率高度依赖草稿-目标的对齐度**：两个模型来自不同训练数据/词表时，草稿几乎没法用。

深度问题：既然验证比生成便宜，能不能递归下去——让草稿模型也用一个更小的模型来投机加速自己（分层投机）？极限情况下，这套"生成难验证易"的思想是否揭示了自回归生成本身的冗余：模型花大力气逐字生成的内容里，有多大比例其实是"低信息量、高度可预测"的、本不需要动用全部参数？

### 8. 和后续知识的连接

- 它直接接续 **KV Cache**（上一课）：草稿和目标模型各自维护 KV Cache，验证被拒后需要正确回滚草稿模型多算的缓存位置。
- 它建立在**自回归生成**与**因果注意力**之上：正因为验证时可以把草稿 token 当作"已知的过去"，因果注意力才能一次前向并行算出所有位置的分布。
- 它和**采样策略（Temperature / Top-k / Top-p）**深度交织：接受-拒绝规则要在这些采样分布上正确定义 $p$ 和 $q$，才能保证等价。
- 往后看，它是理解现代推理引擎（vLLM、TensorRT-LLM）和 **Medusa / EAGLE** 等"并行草稿头"变体的地基，也和**缩放定律**呼应——当推理成本主导总成本时，这类零质损加速直接改变模型部署的经济账。
