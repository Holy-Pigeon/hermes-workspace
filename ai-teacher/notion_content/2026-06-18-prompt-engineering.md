## 今日知识点：Prompt Engineering（提示工程）——不改一个参数，靠"怎么问"撬动大模型的能力

### 所属类目
类目 20：大语言模型应用

### 问题入口
你手上有一个训练好的大语言模型（LLM），参数固定、权重不动。现在要它做一件新任务：把一段英文翻成法文、从客服对话里抽取退款金额、解一道数学应用题。

传统机器学习的答案是：收集这个任务的标注数据，再训练（或微调）一个模型。但这里有个尴尬的现实——

- 你可能只有 3 个例子，凑不出训练集；
- 微调一个百亿参数模型要 GPU、要时间、要工程；
- 而且你今天想翻译，明天想抽取，后天想写代码，难道每个任务都微调一份权重？

**真实问题：能不能不动模型的任何一个参数，仅仅通过"输入文本怎么写"，就让同一个模型去完成千变万化的任务？** 这就是提示工程要回答的问题。

### 朴素想法
最自然的想法是：模型不是会"续写"吗？那我直接把任务描述写进去，让它续写答案。

比如想做情感分类，就输入：

```
这部电影太好看了。情感是：
```

期待模型续写出"正面"。这个想法本身是对的——它正是提示工程的雏形。但朴素地写，会遇到一连串"为什么不灵"的坑。

### 卡住的地方
朴素地"把问题写进去"经常翻车，典型失败场景：

1. **模型不知道你要什么格式**。你问"这部电影太好看了。情感是："，它可能续写出"我看了三遍"而不是"正面"。任务意图没说清。
2. **模型没见过这种问法**。直接发指令"判断情感"，早期的基座模型（base model，只做过"预测下一个词"预训练、没做过指令微调的模型）根本不理解"指令"这个概念，它只会接着你的话往下编。
3. **复杂推理一步到位会错**。问"一个班 23 人，又来了 5 人走了 8 人，现在几人？"，模型可能直接蹦出一个数字，且经常算错——因为它在"一次前向传播"里就要吐出答案，没有中间演算的空间。

这三个坑，分别对应提示工程里三个关键技巧：**清晰的指令、In-context Learning（上下文学习 / 给例子）、Chain-of-Thought（思维链 / 让它一步步想）**。

### 1. 概念

**提示工程（Prompt Engineering）**：在不修改模型权重的前提下，通过精心设计输入文本（即"提示"，prompt），引导大语言模型产生期望输出的方法论。

它的核心机制建立在一个事实上：自回归语言模型本质是在估计条件概率

$$
P(\text{答案} \mid \text{提示}) = \prod_{t=1}^{T} P(y_t \mid y_{<t}, \ \text{prompt})
$$

其中 $y_t$ 是输出的第 $t$ 个 token（词元），$y_{<t}$ 是它前面已经生成的所有 token，$\text{prompt}$ 是你给的输入。**提示工程做的事，就是改变这个条件 $\text{prompt}$，从而改变整个输出分布**——模型没变，但你把它"条件"到了概率空间中更有利的区域。

通俗理解：模型像一个读过海量书、什么都懂一点的"超级实习生"，但它没有自己的目标，只会顺着你给的开头往下接。你给的开头越精准地框定了"现在处于哪种情境"，它接出来的内容就越贴近你要的。但类比到此为止——回到机制：你不是在"哄"它，你是在用文本把它的输出概率分布往某个峰值上推。

三个核心子技巧：

- **Zero-shot（零样本）**：只给指令，不给例子。"把下面这句翻成法语：……"
- **Few-shot / In-context Learning（少样本 / 上下文学习）**：在提示里塞几个"输入→输出"示范，模型从这些例子里现学规律，再套用到新输入。注意：它**没有更新任何权重**，纯靠注意力在上下文里"读"出模式。
- **Chain-of-Thought（CoT，思维链）**：引导模型先写出推理步骤，再给最终答案，把"一步出答案"拆成"多步演算"。

### 2. 推导与证明

**为什么 Few-shot 不更新权重也能"学"？** 这是提示工程最反直觉、也最值得推导的地方。

关键步骤：

1. **预训练目标埋下的种子**。模型在预训练时见过海量形如"模式 A→结果 a，模式 A→结果 b……"的文本（比如词典、问答对、列表）。它学会的不只是具体知识，还有一种元能力：**当上下文里出现重复的结构，就沿着这个结构续写**。这叫"归纳"（induction）能力。

2. **注意力头充当"模式匹配器"**。研究者在小模型里发现了所谓的 induction head（归纳头）：当输入是 `[A][B] … [A]` 时，这种注意力头会去找上一次 `A` 后面跟的是什么（`B`），然后预测当前的 `A` 后面也跟 `B`。把这个机制放大，就是 Few-shot：

$$
\underbrace{x_1 \rightarrow y_1, \ x_2 \rightarrow y_2, \ x_3 \rightarrow y_3,}_{\text{示范}} \ \underbrace{x_{\text{new}} \rightarrow}_{\text{待补全}} \ ?
$$

模型用注意力在前面的示范里找到"输入→输出"这个映射结构，把它"应用"到 $x_{\text{new}}$ 上。整个过程发生在一次前向传播的激活值里，权重 $W$ 始终没变。

3. **CoT 为什么有效——给计算腾出"草稿纸"**。模型的算力受限于它能做的串行计算步数。一次前向传播深度有限，硬要"一步算出"复杂结果，等于要求在固定深度内完成所有运算。而 CoT 让模型把中间结果**写到输出里**，下一步生成时这些中间结果又变成了输入（因为是自回归），相当于把"内存"外置到文本上：

$$
P(\text{答案} \mid \text{问题}) = \sum_{\text{推理链 } r} P(\text{答案} \mid r, \text{问题}) \, P(r \mid \text{问题})
$$

显式生成推理链 $r$，等于让模型沿着一条高概率的中间路径走，每一步只需做一点点局部计算，难度被摊薄了。这就是为什么"Let's think step by step"这一句话能显著提升数学题正确率。

### 3. 来源与历史动机
- 论文一：《Language Models are Few-Shot Learners》（GPT-3 论文）
- 作者：Tom B. Brown 等（OpenAI）
- 时间：2020 年
- 链接：https://arxiv.org/abs/2005.14165

- 论文二：《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》
- 作者：Jason Wei 等（Google）
- 时间：2022 年
- 链接：https://arxiv.org/abs/2201.11903

历史背景：GPT-3（2020）第一次让世界看到，一个 1750 亿参数的模型，**不微调、只给几个例子**，就能在翻译、问答、算术上达到惊人水平。这颠覆了"每个任务都要训练专用模型"的范式，"prompt"一词从此成为核心工程对象。2022 年的 CoT 论文则揭示：模型的推理能力一直都在，只是需要被"问对"才会显现——一句简单的引导语就能解锁，这种"涌现"现象推动了后来对提示设计的系统研究。

### 4. 核心用途

用途一：**快速原型与任务适配**。产品想加一个"自动总结用户评论"功能，不必收集数据训练模型，写一个好的 prompt 当天就能上线验证。提示工程把"做一个 AI 功能"的成本从"几周训练"压到"几小时调 prompt"。

用途二：**复杂推理与 Agent 工作流**。CoT、ReAct（推理+行动）、思维树等都是提示工程的延伸。现代 AI Agent 调用工具、分解任务、自我反思，底层都是结构化的提示设计。

用途三：**对齐与安全护栏**。System prompt（系统提示）给模型设定角色、边界和行为准则（"你是客服助手，不讨论政治"）。这是用提示在推理时施加约束，成本远低于重新训练。

### 5. 反直觉点与常见误区

反直觉点：**Few-shot 里的例子，模型并没有"学会"——它甚至可能不在乎标签对不对**。有研究（Min et al., 2022）发现，把 Few-shot 例子的标签**随机打乱**，模型性能下降有限。这说明示范的主要作用是告诉模型"任务格式长什么样、输出空间是什么"，而不是真的从标签里学到映射。这彻底打破了"给例子=教模型正确答案"的直觉。

常见误区：
- **误区一：以为 prompt 写得越长越详细越好**。过长的提示会稀释关键指令、挤占上下文窗口，甚至让模型抓错重点。清晰 > 冗长。
- **误区二：以为提示工程能突破模型的能力上限**。prompt 只能"激发"模型已有的能力，不能凭空创造它没有的知识。一个没学过某冷门语言的模型，再好的 prompt 也翻不出来。提示工程是"解锁"，不是"赋能"。
- **误区三：把 Few-shot 的"学习"理解成训练**。它不更新任何权重，换一个对话就忘光了，和真正的 Fine-tuning 有本质区别。

### 6. 小实验

不用写代码，找任意一个聊天大模型，做两组对比：

**实验 A（CoT 的威力）**：
- 先问：`一个水池有 3 个进水管，单独开分别 6/4/3 小时灌满，三管齐开几小时灌满？直接给答案。`
- 再问同样的题，但结尾加一句：`请一步步列出计算过程，最后再给答案。`

对比两次答案，你大概率会看到：要求"一步步"的那次，正确率明显更高、过程更可核查。这就是把中间计算"写到草稿纸上"的效果。

**实验 B（Few-shot 框定格式）**：
- 直接问：`苹果` —— 模型会自由发挥。
- 改成给两个示范：`狗→动物；玫瑰→植物；苹果→` —— 模型几乎一定续写"水果"或"植物"。

你没训练它，只是用上下文把它的输出"框"进了一个分类任务。亲手感受一次"权重没变、行为却变了"。

### 7. 批判性思考

局限性：
- **脆弱性**。同一个意思换个措辞，输出可能天差地别；模型对提示的格式、措辞、甚至例子顺序都敏感。这让 prompt 难以保证稳定，工业界因此发展出 prompt 的版本管理和自动评测。
- **不可解释**。为什么这个 prompt 比那个好，往往没有理论保证，大量靠试错。提示工程至今仍有"炼丹"色彩。
- **能力天花板**。如前所述，prompt 无法突破模型本身的知识与推理边界。

深度问题：如果 Few-shot 的标签对错都无所谓，那"In-context Learning"到底是不是"学习"？它和梯度下降式的学习有没有数学上的等价关系？（提示：有研究尝试证明，Transformer 在前向传播中模拟了一步梯度下降——这是 2023 年的热点话题，值得深挖。）

### 8. 和后续知识的连接

- 提示工程是不动权重的"轻量适配"；下一课的 **Fine-tuning（微调）** 则是真正更新权重的"重量适配"。两者是 LLM 落地的两条主线，理解了 prompt 的边界，才知道何时该上微调。
- CoT 里"让模型一步步想"的思想，是后续 **RLHF**（用人类偏好训练模型更会推理、更对齐）和各种 reasoning model（推理模型）的前身。
- Few-shot 依赖把信息塞进上下文，这直接通向 **RAG（检索增强生成）**——与其指望模型记住一切，不如在 prompt 里实时检索并塞进相关资料。RAG 本质就是"自动化的、带外部知识的 Few-shot"。
- 而这一切之所以能工作，全靠前面学过的 **自回归生成、因果注意力、缩放点积注意力**：正是注意力机制让模型能"读懂"上下文里的示范和指令。提示工程是站在整个 Transformer 机制肩膀上的应用层魔法。
