## 今日知识点：PagedAttention（分页注意力）——像操作系统管内存那样管 KV Cache

### 所属类目
类目 19：高效注意力

### 问题入口
我们已经学过 **KV Cache（键值缓存）**：自回归生成时，把每个已生成 token 的 Key 和 Value 存起来，下一步就不用把整段前文重算一遍。这让单条请求的解码从 $O(n^2)$ 的重复计算降成每步 $O(n)$。

但当你真正把一个大模型**部署成线上服务**、同时伺候几十上百个用户时，一个新的瓶颈冒出来了：**显存（GPU memory）不够用，而且用得极其浪费**。

具体问题是：一条请求的 KV Cache 要占多少显存？它随着"这条对话生成到多长"不断增长，而你**事先根本不知道它会生成多长**。有的用户问一句就结束，有的让模型写一篇 2000 字的文章。你该给每条请求预留多少显存？这个"预留多少"的决定，直接决定了你的 GPU 能同时服务多少用户——也就是吞吐量（throughput）。PagedAttention 就是来解决这个"显存怎么分"的问题的。

### 朴素想法
最自然的做法：**给每条请求预留一整块连续的显存**，大小按"最大可能长度"算。

比如模型最大支持 2048 个 token，那我就给每条请求预留能装下 2048 个 token 的 KV Cache 的连续空间。为什么要"连续"？因为注意力计算时要对一整段 Key/Value 做矩阵运算，GPU 上连续内存的读取最快，代码也最好写——一个起始指针加一个长度就能索引整段缓存。

这套逻辑简单直接，早期的推理框架基本都这么干。

### 卡住的地方
"预留最大长度的连续显存"会在两个地方严重浪费显存，浪费到令人发指：

1. **内部碎片（internal fragmentation）**：你给每条请求预留了 2048 个 token 的空间，但用户实际只生成了 30 个 token。剩下 2018 个 token 的空间**被这条请求霸占着、却一直空着**，别的请求也用不了。研究测量发现，这种预留式方案下，真正被有效利用的显存**常常不到 40%**——一大半 GPU 显存在空转。

2. **外部碎片（external fragmentation）**：不同请求要的连续块大小不一，请求来来走走，显存里被切成一堆大小不齐、又不相邻的空洞。就算总的空闲显存加起来够装下一条新请求，也可能因为**找不到一整块足够大的连续空间**而不得不拒绝它。

3. **无法共享**：多个请求如果共享同一段前缀（比如同一个 system prompt、或者对同一个问题采样多个答案），这些完全相同的 KV Cache 被复制了好多份，各占各的连续块，白白浪费。

根子在于两个字：**连续**。一旦坚持"每条请求一块连续显存 + 按最坏情况预留"，浪费就是结构性的、躲不掉的。

### 1. 概念
**PagedAttention** 的核心思想：**放弃"KV Cache 必须连续存放"这个假设**，改成把 KV Cache 切成固定大小的小块（block），让这些块**在物理显存里可以东一块西一块地散落**，再用一张"目录表"记住每条请求的块都散在哪儿。

这个想法直接抄自操作系统的**虚拟内存分页（paging）**：

- 操作系统不要求一个程序的内存在物理内存里连续。它把内存切成固定大小的**页（page）**，程序看到的是连续的"逻辑地址"，硬件通过一张**页表（page table）**把逻辑地址翻译成散落各处的物理地址。
- PagedAttention 如法炮制：每条请求的 KV Cache 被切成固定大小的 **KV block**（比如每块存 16 个 token 的 Key/Value）。每条请求维护一张 **block table（块表）**，记录它逻辑上的第 0、1、2……块，分别对应物理显存里的哪一块。

**通俗理解**：从前是"给每位客人预订一整排连号的座位，还按最多可能来的人数订"，空座全浪费。现在是"来一个人给一个座，坐哪不要紧，前台拿个小本子记下每桌的人分别坐在哪几个散座"。座位利用率立刻拉满。——但类比到此为止，回到机制：关键在于注意力计算**不需要 Key/Value 物理连续**，只要能按块把它们取出来做点积就行，所以"打散 + 用块表索引"这套操作完全可行。

正式一点，标准注意力对一条长为 $n$ 的序列计算：

$$
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{Q K^\top}{\sqrt{d}}\right) V
$$

其中 $Q$ 是当前 token 的 query 向量，$K = [k_1, k_2, \dots, k_n]$ 和 $V = [v_1, v_2, \dots, v_n]$ 是前文所有 token 的 key/value，$d$ 是每个头的维度，除以 $\sqrt{d}$ 是我们学过的缩放。PagedAttention 只是把 $K$、$V$ 按每 $B$ 个 token 切成若干块（$B$ 是块大小，如 16），逐块从散落的物理显存里取来参与 $Q K^\top$ 与乘 $V$ 的运算，**数学结果和连续存放时一模一样**，改变的只是"数据放在哪、怎么取"。

### 2. 推导与证明
PagedAttention 不是一个新的数学公式，而是一次**存储与索引机制的重构**，其"正确性"在于证明"打散存放不改变注意力结果"，"有效性"在于算清它省了多少显存。我们分两步走。

关键步骤：

1. **第一步：确认注意力对 KV 的存放顺序/位置不敏感。**
   注意力的核心是 $\text{softmax}(Q K^\top / \sqrt{d}) V$。展开看，输出是对所有 value 的加权和：

   $$
   o = \sum_{j=1}^{n} \alpha_j v_j, \qquad \alpha_j = \frac{\exp(q \cdot k_j / \sqrt{d})}{\sum_{t=1}^{n} \exp(q \cdot k_t / \sqrt{d})}
   $$

   这里 $\alpha_j$ 是第 $j$ 个 token 分到的注意力权重，$q \cdot k_j$ 是点积。注意：只要每个 $k_j$ 都能和它对应的 $v_j$ 配对取到，**它们在物理显存里存在哪个地址、是否相邻，完全不影响 $o$ 的值**。求和是可交换的。这就是"允许打散"的理论依据——为什么要先证这一步？因为如果打散会改变结果，那再省显存也是错的。

2. **第二步：把连续存储换成"块表 + 按需分配"。**
   为每条请求维护一张 block table。逻辑块号 $\to$ 物理块号 的映射由这张表负责。生成新 token 时：当前最后一个块没满，就填进去；填满了，才**按需（on-demand）**新分配一个物理块并登记到块表。为什么要"按需"而不是"预留最大长度"？因为这样一条请求**用多少显存就占多少**，不再为"可能生成到 2048"而预扣空间——内部碎片几乎归零。

3. **第三步：算账，看碎片降到多少。**
   固定块大小为 $B$（如 16）。一条实际长度为 $L$ 的序列需要 $\lceil L / B \rceil$ 个块。**唯一被浪费的显存**只剩最后一个没填满的块里的空位，最多浪费 $B - 1$ 个 token 的空间。于是浪费比例上界约为 $\frac{B-1}{L}$：当 $L$ 远大于 $B$ 时趋近于 0。对比朴素方案"预留 2048、实际用 30"浪费掉 $2018/2048 \approx 98.5\%$，PagedAttention 用 $B=16$、$L=30$ 时最多浪费 1 个块里的空位（$< 16/30$，且实践中被大量请求摊薄），显存有效利用率从不到 40% 提到接近 96%+。为什么这一步重要？它把"打散"这个定性想法，变成了"利用率提升一倍多"的定量收益——这才是它值得做的理由。

4. **第四步（可选增益）：前缀共享（copy-on-write）。**
   块表带来一个免费红利：如果两条请求共享前缀（同一 system prompt，或对同一 prompt 采样多个输出），它们的 block table 可以**指向同一批物理块**，只读共享、不复制。等某条请求要在共享块上写新内容时，才复制那一块（写时复制，copy-on-write）——又是直接借用操作系统的成熟机制。这让并行采样、beam search 的显存开销大幅下降。

### 3. 来源与历史动机
- 论文：《Efficient Memory Management for Large Language Model Serving with PagedAttention》
- 作者：Woosuk Kwon, Zhuohan Li, Siyuan Zhuang, Ying Sheng 等（UC Berkeley，Sky Computing Lab / vLLM 团队）
- 时间：2023 年 9 月（SOSP 2023）
- 链接：https://arxiv.org/abs/2309.06180

历史背景：2023 年大模型推理服务需求爆发，GPU 显存成了最贵、最紧的资源。当时主流框架（如早期的 HuggingFace Transformers 生成、FasterTransformer）都用"预留连续 KV Cache"的方案，测量发现显存利用率常常只有 20%~40%，大量算力被碎片拖累。PagedAttention 团队敏锐地发现，这和 1960 年代操作系统面对的"物理内存碎片"是**同一个问题**，于是把经典的虚拟内存分页搬了过来。基于它构建的推理引擎 **vLLM** 相比当时的 HuggingFace 方案吞吐量提升可达 **数倍到十几倍**，迅速成为工业界大模型部署的事实标准之一。这是"用一个老领域的成熟思想解决新领域的新问题"的经典范例。

### 4. 核心用途
用途一：**大模型推理服务的高吞吐部署**。这是 PagedAttention 的主场。它是 vLLM 的核心机制，让同一张 GPU 能同时容纳更多并发请求（更大的 batch），直接提升每秒能服务的 token 数。今天绝大多数在线 LLM API 的后端都用了 PagedAttention 或其衍生方案。

用途二：**并行采样与搜索的显存复用**。当你对一个 prompt 采样多个候选答案（如 $n=8$ 采样、beam search），这些候选共享 prompt 的 KV Cache。前缀共享让它们复用同一批物理块，显存开销从"复制 8 份"降到"1 份 + 少量增量"。

用途三：**长 system prompt / 多轮对话场景**。很多应用有很长的固定 system prompt（几百上千 token）。多个用户请求共享这段前缀时，其 KV block 只存一份、被多张块表指向，省下的显存相当可观（这也催生了后来的 "prefix caching" 优化）。

### 5. 反直觉点与常见误区
反直觉点：**PagedAttention 一点计算量都没省，却让服务快了好几倍。** 它和 FlashAttention 是两类完全不同的优化——FlashAttention 省的是显存**读写次数（IO）**，PagedAttention 省的是显存**占用与碎片**。前者让单次注意力算得快，后者让 GPU 能同时装下更多请求。吞吐量的提升几乎全部来自"batch 变大了"，而不是"每次算得快了"。理解这一点，才能分清各种"高效注意力"到底在优化哪个维度。

常见误区：
- **误区一：以为 PagedAttention 改变了注意力的数学。** 完全没有。输出结果和连续存放逐比特相同，它只动了 KV Cache 的"存储布局与索引方式"，属于系统/内存管理层面的创新，不是新的注意力算子。
- **误区二：把 PagedAttention 和 FlashAttention 当成竞品、二选一。** 恰恰相反，它们正交且互补，vLLM 里两者同时用：FlashAttention 加速块内注意力计算，PagedAttention 管块的分配与调度。一个管"算得快"，一个管"装得下"。

### 6. 小实验
不用 GPU，用一张纸就能体会碎片问题。

假设 GPU 显存能装下 100 个 token 的 KV Cache，块大小 $B = 10$（每块 10 个 token）。现在来 5 条请求，实际生成长度分别是 8、35、3、12、20 个 token。

- **朴素方案（每条预留最大长度 50）**：第一条就要预留 50，第二条又要 50，总共需要 100 才能装下**两条**，第 3、4、5 条只能排队等待。有效利用：真正用到的是 $8 + 35 = 43$，利用率 $43/100 = 43\%$。
- **PagedAttention（按 $B=10$ 分块、按需分配）**：各请求需要的块数为 $\lceil 8/10\rceil=1$、$\lceil 35/10\rceil=4$、$\lceil 3/10\rceil=1$、$\lceil 12/10\rceil=2$、$\lceil 20/10\rceil=2$，共 $1+4+1+2+2 = 10$ 块 = 100 个 token 的空间，**五条请求全部同时装下**。有效利用：实际用到 $8+35+3+12+20 = 78$ 个 token，利用率 $78/100 = 78\%$，浪费只来自每条最后一个没填满的块。

心算一下就懂了：同一张卡，朴素方案同时只能跑 2 条，PagedAttention 能跑 5 条——吞吐量 2.5 倍的差距，全来自"别再按最坏情况预留连续显存"。

### 7. 批判性思考
局限性：
- **块表带来间接寻址开销**：每次取 KV 要先查块表再访问物理块，比"一个指针直读连续内存"多一层跳转。这需要精心写的 CUDA kernel 才能把开销压到可忽略——机制简单，工程实现并不简单。
- **块大小 $B$ 是个权衡**：$B$ 太小，块表变大、管理开销上升；$B$ 太大，最后一块的内部碎片又回来了。需要按场景调。
- **它解决的是"显存怎么分"，不解决"KV Cache 本身太大"**。序列极长时，KV Cache 的总量依然是硬约束，这要靠 MQA/GQA（我们学过）压缩每层 KV、或量化 KV Cache 等手段配合。

深度问题：PagedAttention 把操作系统的"虚拟内存分页"整套搬到了 GPU 显存管理。那么操作系统里另外几个成熟机制——**换页到磁盘（swap，即把冷数据挪到 CPU 内存/硬盘）**、**页面置换算法（LRU 等）**——能不能也搬过来管 KV Cache？（提示：vLLM 后续确实做了 KV Cache 的 CPU offload 与抢占重计算。）当一个领域的问题和另一个成熟领域"同构"时，你能顺着这个映射预测出多少还没被实现的优化？这正是"抽象迁移"的威力。

### 8. 和后续知识的连接
- **向前接**：它是我们学过的 **KV Cache** 的直接延伸——KV Cache 解决"别重算前文"，PagedAttention 解决"这些缓存在显存里怎么放才不浪费"。它也和 **MQA / GQA / 滑动窗口注意力** 属于同一条"让注意力在真实系统里跑得起、跑得省"的主线：GQA 减少每 token 的 KV 大小，PagedAttention 优化 KV 的存放，两者叠加。
- **向后接**：理解 PagedAttention 是读懂现代 LLM **推理引擎（vLLM、TensorRT-LLM、SGLang 等）**的钥匙。后续话题如 **prefix caching（前缀缓存）**、**continuous batching（连续批处理）**、**KV Cache 量化 / offload**，都建立在"KV Cache 是可分块、可共享、可调度的资源"这一 PagedAttention 奠定的视角之上。当你之后思考"一个大模型服务的成本和速度由什么决定"时，这套内存管理思想会是核心答案之一。
