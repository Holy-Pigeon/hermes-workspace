## 今日知识点：Sparse Attention（稀疏注意力）——高效注意力的基础思想

### 所属类目
类目 19：高效注意力

### 问题入口
标准 Transformer 的自注意力让序列里**每个位置都和其他所有位置**两两交互。序列长度 $n$ 时，注意力矩阵是 $n\times n$，计算和显存都是 $O(n^2)$。处理一篇 1000 词的文章还行，可一旦要处理一本书（几万 token）、一段高分辨率图像（上万像素）、或一段长音频，$n^2$ 就会爆炸：$n=10000$ 时注意力矩阵有 1 亿个元素，$n=100000$ 时是 100 亿。显存撑不住、速度慢到不可接受。怎么让注意力处理超长序列，又不至于被 $n^2$ 拖垮？

### 朴素想法
最直接的想法是：**截断**——超过 512 或 1024 token 就把多余的部分丢掉，或者切成互不相干的小块分别处理。再朴素一点：把序列**降采样**，每隔几个 token 取一个，硬把 $n$ 变小。

### 卡住的地方
截断和分块会**切断长距离依赖**。书的第 300 页可能要呼应第 5 页的伏笔，切成互不相干的块后，模型永远看不到跨块的联系。降采样则直接丢失细节，分辨率没了。这些办法都是在"要么算全 $n^2$、要么放弃远处信息"之间二选一。

真正卡住人的疑问是：**全连接注意力真的有必要吗？** 经验告诉我们，一个 token 真正强相关的，往往只是**少数几个**位置（邻近的词、句法上相关的词、几个关键的远处锚点），而不是全部 $n$ 个。换句话说，那个 $n\times n$ 的注意力矩阵实际上是**高度稀疏**的——大部分权重接近 0。既然如此，我们为什么要花 $O(n^2)$ 去算一个本质上稀疏的东西？

### 1. 概念
**直觉**：不让每个位置都去看所有位置，而是**只看一个精心设计的子集**。每个位置只和"该看的那些"位置算注意力，其余直接跳过（视为权重 0）。这样把稠密的 $n\times n$ 交互变成稀疏的，计算量从 $O(n^2)$ 降到 $O(n\sqrt{n})$ 甚至 $O(n\log n)$、$O(n)$。

**通俗类比**：开大会发言。全连接注意力像"每个人都要和会场里其他每个人单独私聊一遍"——人一多就瘫痪。稀疏注意力像"每个人只和邻座聊（局部），外加每个人都和主持人聊（全局枢纽），主持人再把信息转达全场"。少数关键连接就能让信息传遍全场。但类比到此为止——关键在于"该看哪些位置"这套**注意力模式（attention pattern）**怎么设计才能既省算力、又不丢信息。

**正式定义**：标准注意力中位置 $i$ 关注全集 $\{1,\dots,n\}$。稀疏注意力为每个位置 $i$ 指定一个**注意力集合** $S_i \subseteq \{1,\dots,n\}$，且 $|S_i| \ll n$，只在该集合上计算：
$$\text{Attention}(Q,K,V)_i = \sum_{j \in S_i} \frac{\exp(q_i \cdot k_j / \sqrt{d})}{\sum_{k \in S_i}\exp(q_i \cdot k_k/\sqrt{d})}\, v_j$$
不同的稀疏注意力方法，本质区别就在于**如何设计 $S_i$**。

### 2. 推导与证明
我们从复杂度入手，理解稀疏化为什么有效、以及好的稀疏模式要满足什么条件。

**第 1 步：标准注意力的复杂度来源。**
$$\text{Attention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d}}\right)V$$
其中 $Q,K,V \in \mathbb{R}^{n\times d}$。瓶颈是 $QK^T$：它是 $n\times n$ 矩阵，需要 $n^2$ 次内积，**时间和显存都是 $O(n^2 d)$**。$n$ 一大就崩。

**第 2 步：把全连接换成稀疏连接。**
若每个位置只关注 $|S_i|=s$ 个位置，则总计算量降为 $O(n\cdot s\cdot d)$。当 $s$ 是常数时为 $O(n)$，当 $s=O(\sqrt n)$ 时为 $O(n\sqrt n)$。**为什么能省**：我们干脆不计算那些注定权重接近 0 的内积。

**第 3 步：稀疏模式必须保证"信息可达"。**
关键约束：随便挖空连接会切断信息流。一个好的稀疏模式要保证**任意两个位置都能在有限步内通过注意力路径互相影响**（图论上即连通、且直径小）。Sparse Transformer 用两种模式的组合达成这一点：
- **局部/滑窗注意力（local / sliding window）**：位置 $i$ 关注它前后窗口内的若干邻居。**为什么**：相邻 token 通常最相关，捕捉局部依赖，复杂度 $O(n\cdot w)$（$w$ 为窗口宽）。
- **跨步/全局注意力（strided / global）**：位置 $i$ 额外关注每隔固定步长的位置，或设少数"全局枢纽"位置让所有人都能连到。**为什么**：提供长距离捷径，让信息两三跳就能跨越全序列。

**第 4 步：复杂度论证。**
Sparse Transformer 把注意力分解成上述两类，每类约 $O(n\sqrt n)$，组合后整体 $O(n\sqrt n)$。直觉上：第一类管"近处密集联系"，第二类管"远处稀疏捷径"，两者叠加既覆盖局部又保留全局可达性。**为什么 $\sqrt n$**：把序列想成 $\sqrt n \times \sqrt n$ 的网格，一个跨度负责"行内"、一个负责"列内"，任意两点经"先走到同列、再走到同行"两跳可达，每跳代价 $O(\sqrt n)$。

**第 5 步：因果性与正确性。**
在自回归生成中，$S_i$ 只能取 $j \le i$（不能看未来）。只要稀疏模式在因果掩码下仍连通，模型就能在 $O(n\sqrt n)$ 内近似全连接的表达力。Child 等人实验证明，这种近似在长序列任务上几乎不损失质量，却大幅降低算力。

### 3. 来源与历史动机
- 论文：Generating Long Sequences with Sparse Transformers
- 作者：Rewon Child, Scott Gray, Alec Radford, Ilya Sutskever（OpenAI）
- 时间：2019 年
- 链接：arXiv:1904.10509 （https://arxiv.org/abs/1904.10509）

**历史背景**：2019 年，Transformer 已是 NLP 主流，但 $O(n^2)$ 让它难以建模长序列（长文档、高分辨率图像、音频）。OpenAI 的 Sparse Transformer 是最早系统性提出"用结构化稀疏模式把注意力降到 $O(n\sqrt n)$"的代表作之一，并成功把 Transformer 用到了数万长度的序列（图像、音频生成）。它开启了"**高效注意力（efficient attention）**"这一整条研究脉络，后续涌现出 Longformer、BigBird、Reformer、Performer 等一大批工作。值得一提，论文作者中的 Radford 和 Sutskever 正是 GPT 系列的核心人物——对长序列的渴求贯穿了大模型的发展。

### 4. 核心用途
- **用途一：长文档/长上下文建模**。书籍级文本、长代码文件、检索增强的大段上下文，稀疏注意力让模型一次性"看得更长"而不爆显存。
- **用途二：高分辨率图像与音频生成**。把图像像素、音频采样点展平成超长序列时，稀疏注意力是让自回归 Transformer 可行的关键（Sparse Transformer 原文就用于图像和音频生成）。
- **用途三：降低长上下文 LLM 的推理成本**。现代支持几十万 token 上下文的模型，常借助稀疏/滑窗注意力的思想压低长序列下的计算与显存开销。

### 5. 反直觉点与常见误区
**反直觉点**：**少看不一定看得差，甚至可能看得更好。** 把全连接砍成稀疏，模型质量却几乎不降——因为真实的注意力矩阵本就高度稀疏，全连接里大部分算力花在了权重接近 0 的"无效内积"上。稀疏化等于把算力集中到真正重要的连接上，是一种"有益的归纳偏置（inductive bias）"。

**误区一：以为稀疏注意力会切断长距离依赖。** 设计良好的稀疏模式（局部 + 全局/跨步）通过"少数捷径"保证任意两位置仍能在常数跳内互相影响，长依赖**没有**丢失，只是从"一步直达"变成"两三跳可达"。

**误区二：把稀疏注意力和"低秩/线性注意力"混为一谈。** 二者都是高效注意力，但思路不同：稀疏注意力是**挖掉部分连接**（注意力矩阵稀疏化）；线性注意力（如 Performer）是用**核函数近似 softmax**、改变计算顺序把复杂度降到 $O(n)$，注意力矩阵仍是"稠密"的只是不显式构造。机制不可混淆。

### 6. 小实验
用一个小矩阵直观感受"稀疏掩码如何省算力又保留主要信息"。我们造一个滑窗（每个位置只看自己和左右各 1 个邻居）掩码，对比稠密注意力：

```python
import numpy as np

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

n, d = 6, 4
np.random.seed(0)
Q = np.random.randn(n, d)
K = np.random.randn(n, d)
V = np.random.randn(n, d)

scores = Q @ K.T / np.sqrt(d)          # n x n 打分

# 稠密注意力
dense = softmax(scores) @ V

# 稀疏：滑窗，每个位置只看 |i-j|<=1，其余置 -inf
mask = np.abs(np.subtract.outer(range(n), range(n))) <= 1
sparse_scores = np.where(mask, scores, -1e9)
sparse = softmax(sparse_scores) @ V

print("被实际计算的连接数  稠密:", n*n, " 稀疏:", int(mask.sum()))
print("两种输出的平均差异:", round(np.abs(dense - sparse).mean(), 3))
```

你会看到稠密用了 36 个连接，稀疏只用 16 个（约 44%），而输出差异不大——局部信息几乎完好保留。把 `n` 调大，稠密的连接数按 $n^2$ 暴涨，滑窗只按 $n\cdot w$ 线性增长，省的比例越来越夸张。这就是稀疏注意力的核心账。

### 7. 批判性思考
**局限性**：
1. **稀疏模式是人手工设计的**。局部 + 跨步等模式是先验假设，未必匹配每个任务真正的依赖结构；选错模式可能漏掉关键连接。
2. **硬件不友好**。非规则的稀疏访存在 GPU 上效率往往打折，理论 FLOPs 省了，实际加速可能没那么多，需要专门的稀疏 kernel。
3. **静态 vs 动态之争**。固定稀疏模式无法因输入自适应；后来的工作（如基于哈希/路由的稀疏）试图动态选择 $S_i$，但带来额外开销与复杂度。

**深度问题**：稀疏注意力的前提是"注意力矩阵本就稀疏"。但如果某个任务**真的需要稠密的全局交互**（每个 token 都和很多远处 token 强相关），强行稀疏化就会损害性能。那么——我们怎么**事先判断**一个任务/一段输入需要多稠密的注意力？有没有办法让模型**自己学**该稠密还是该稀疏，而不是人来拍板？这正是高效注意力领域至今活跃的开放问题。

### 8. 和后续知识的连接
- **Longformer / BigBird**：把"局部窗口 + 全局枢纽 + （随机）稀疏"模式发扬光大，并从理论上证明 BigBird 的稀疏注意力是图灵完备、能逼近全注意力。
- **线性注意力（Performer、Linear Attention）**：另一条降复杂度路线，用核近似把 softmax 注意力变成 $O(n)$，与稀疏注意力互为补充。
- **FlashAttention**：不改稀疏结构，而是用 IO 感知的算法把标准注意力算得又快又省显存——和稀疏化是"正交"的两种高效手段，常配合使用。
- **长上下文 LLM**：今天动辄几十万 token 上下文的大模型，背后都站着稀疏/滑窗/高效注意力的思想。理解 Sparse Attention，就握住了"如何让 Transformer 看得更长"这条主线的起点。
