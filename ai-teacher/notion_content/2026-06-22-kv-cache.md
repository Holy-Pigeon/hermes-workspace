## 今日知识点：KV Cache（键值缓存）——别让模型每吐一个字都把前文重算一遍

### 所属类目
类目 18：Decoder-only 模型

### 问题入口
我们已经学过自回归生成：模型一次只吐一个词，再把这个词接回输入，重新跑一遍，预测下一个词。听起来很合理，但藏着一个要命的浪费。

假设模型已经生成了 "今天 天气 真"，现在要预测第 4 个词。按照自回归的字面流程，我们要把 "今天 天气 真" 这 3 个词整段重新喂进 Transformer，算一遍注意力，得到最后一个位置的输出，再预测 "好"。

下一步预测第 5 个词时，又要把 "今天 天气 真 好" 这 4 个词**整段**重新喂一遍。第 100 个词时，要把前 99 个词整段重算。

问题来了：**前面那些词的内部计算，每一步都在重复，而且重复的量越来越大。** 生成一句 1000 词的话，第 1 个词被重新参与计算了将近 1000 次。这合理吗？有没有办法让"已经算过的东西"不再重算？这就是 KV Cache 要解决的问题。

### 朴素想法
最朴素的做法就是上面描述的：**每生成一个新词，就把"提示词 + 已生成的所有词"拼成完整序列，从头跑一遍完整的前向传播。** 代码写起来最简单——反正模型接受一个序列，输出每个位置的预测，我取最后一个位置就行。

这个想法没有逻辑错误，结果是完全正确的。它只是慢，慢得离谱。

### 卡住的地方
卡在**重复计算的总量随序列长度平方增长**。

回忆缩放点积注意力：对长度为 $n$ 的序列，要算 $n \times n$ 的注意力矩阵。如果每生成一个词都重跑整段：
- 生成第 1 个词：算长度 1 的注意力，约 $1^2$ 的代价
- 生成第 2 个词：算长度 2 的注意力，约 $2^2$
- ……
- 生成第 $n$ 个词：约 $n^2$

总代价是 $1^2 + 2^2 + \dots + n^2 \approx \frac{n^3}{3}$。生成一句话的总计算量是序列长度的**三次方**。序列翻一倍，计算量变 8 倍。这就是为什么朴素自回归生成长文本会越来越卡。

但这里有个关键观察：在因果注意力（causal attention）下，**第 $i$ 个位置的词只能看到它自己和前面的词，看不到后面的。** 这意味着已经生成的词，它们的内部表示在后续步骤里**根本不会改变**——后面新来的词不影响前面词的 Key 和 Value。既然不变，为什么要重算？

### 1. 概念
**KV Cache（键值缓存）**：在自回归生成时，把每个已处理位置算出的 **Key 向量**和 **Value 向量**存进一块缓存里。生成新词时，只需要为**这一个新词**计算它的 Query、Key、Value，然后让新词的 Query 去和"缓存里所有旧 Key + 新 Key"做注意力，对"缓存里所有旧 Value + 新 Value"做加权求和。旧词的 K、V 直接从缓存读，不重算。

通俗理解：想象你在做一场连续的会议记录，每来一位发言人（新 token），你不需要把前面所有人的发言重新听一遍——你手边有一本"发言摘要本"（缓存），新人发言时你只把他的话整理进本子，然后让他参考整本摘要做回应。摘要本就是 KV Cache。

但类比要回到机制。回忆自注意力对单个 Query 的计算：

$$
\text{Attention}(q, K, V) = \text{softmax}\left(\frac{q K^\top}{\sqrt{d}}\right) V
$$

这里 $q$ 是当前新词的 Query 向量（$1 \times d$），$K$ 是所有位置的 Key 堆成的矩阵（$n \times d$），$V$ 是所有位置的 Value（$n \times d$），$d$ 是每个头的维度，$\sqrt{d}$ 是缩放因子。关键点：**算第 $n$ 个词的输出，只需要它自己的 $q$，但要用到全部 $n$ 个 $K$ 和 $V$。** 而这 $n$ 个 K、V 里，前 $n-1$ 个在上一步就已经算过了。KV Cache 就是把它们存下来。

注意：被缓存的只有 **K 和 V，不缓存 Q**。因为旧词的 Query 在后续步骤里再也用不到了——每一步只有"当前新词"需要发出 Query 去查询，旧词不再发问。

### 2. 推导与证明
我们来证明"缓存 K、V 不会改变结果"，并算出省了多少。

关键步骤：

1. **因果掩码保证旧词表示不变。** 在因果注意力里，位置 $i$ 的输出只依赖位置 $1$ 到 $i$ 的输入。当我们新增位置 $n+1$ 时，它的存在不会进入位置 $1, \dots, n$ 的注意力计算（它们看不到未来）。所以位置 $1, \dots, n$ 的 Key、Value 在加入新词后**逐元素完全相同**。既然数值不变，缓存它们 = 重算它们，结果一致。这就证明了 KV Cache 不改变输出，是精确优化，不是近似。

2. **每步的增量计算。** 用缓存后，生成第 $n$ 个词只需要：为新词算一组 $q, k, v$（代价约 $d^2$，与 $n$ 无关），把 $k, v$ 追加进缓存，然后做一次 $q$ 对 $n$ 个 K 的点积（代价约 $n \cdot d$）。单步代价约正比于 $n$。

3. **总代价从三次方降到二次方。** 全部 $n$ 步加起来：$1 + 2 + \dots + n \approx \frac{n^2}{2}$。对比朴素法的 $\frac{n^3}{3}$，省掉了一整个数量级（一个 $n$ 因子）。这就是为什么所有 LLM 推理引擎默认都开 KV Cache。

4. **代价是显存。** 缓存要存下每一层、每个头、每个位置的 K 和 V。缓存大小约为：

$$
\text{Cache} = 2 \times L \times n \times d_{model} \times b
$$

其中 $L$ 是层数，$n$ 是当前序列长度，$d_{model}$ 是隐藏维度，$b$ 是每个数值的字节数（如 FP16 是 2 字节），前面的 $2$ 是 K 和 V 各一份。这个量随序列长度**线性增长**——序列越长，缓存吃的显存越多。这也是长上下文推理的主要显存瓶颈。

### 3. 来源与历史动机
- 论文：KV Cache 不是某一篇论文"发明"的独立概念，它是 Transformer 自回归解码的工程标配，最早伴随 "Attention Is All You Need"（Vaswani et al., 2017）的解码器设计自然出现，并在后续推理系统中被明确命名和优化。
- 关键相关工作：Shazeer 的 "Fast Transformer Decoding: One Write-Head is All You Need"（2019，提出 Multi-Query Attention 压缩 KV Cache）；vLLM 的 "Efficient Memory Management for Large Language Model Serving with PagedAttention"（Kwon et al., 2023）。
- 时间：2017 概念雏形，2019 起成为优化热点，2023 后随长上下文和高并发服务成为核心议题。
- 链接：https://arxiv.org/abs/1911.02150 （Multi-Query Attention）、https://arxiv.org/abs/2309.06180 （PagedAttention / vLLM）

历史背景：当 GPT 类模型开始做实时对话和长文生成，"每步重算"的三次方代价让推理慢到无法商用。KV Cache 把它降到二次方，是让 LLM 能交互式响应的基础。但缓存又带来显存爆炸的新问题，于是催生了 MQA、GQA、PagedAttention 等一系列压缩与管理技术。

### 4. 核心用途
用途一：**LLM 推理加速（最核心）。** 你在 ChatGPT 里看到文字一个个蹦出来，背后就是开着 KV Cache 的自回归解码。没有它，长回答的延迟会随长度三次方爆炸，无法实时。

用途二：**长上下文与多轮对话的状态保持。** 多轮对话里，前面所有轮次的 K、V 可以一直留在缓存中，新一轮只增量计算用户的新输入，历史不必重算。这也是为什么对话历史越长，显存占用越高、有时会被截断。

用途三：**推动架构创新的约束来源。** 正因为 KV Cache 是显存大户，才有了 Multi-Query Attention（多头共享一份 K、V）、Grouped-Query Attention（分组共享，GQA，被 LLaMA-2、Mistral 采用）来把缓存压小几倍。理解 KV Cache 是理解这些现代架构选择的前提。

### 5. 反直觉点与常见误区
反直觉点：**KV Cache 是一个"精确"优化，不是"近似"。** 很多加速技巧（量化、剪枝、稀疏注意力）是用一点精度换速度。但 KV Cache 的输出和朴素重算**逐比特意义上等价**（忽略浮点累加顺序的微小差异），因为因果掩码保证了旧词表示本来就不变。它纯粹是消除冗余，不损失任何质量。

常见误区：
- **误区一：以为 Q 也要缓存。** 不缓存 Q。每一步只有当前新词需要发出 Query，旧词的 Query 用完即弃，缓存它毫无意义。只缓存 K 和 V。
- **误区二：以为 KV Cache 能省显存。** 恰恰相反，它是**用显存换计算**——多花了一大块显存来存 K、V，换取不重算。在长上下文场景，KV Cache 往往是显存的头号消耗者，甚至超过模型权重本身。
- **误区三：以为训练时也用 KV Cache。** 训练时是把整段序列一次性并行喂入、一次算完所有位置（teacher forcing），不存在"一个个生成"的过程，因此**训练阶段没有 KV Cache**。它只在推理/生成阶段存在。

### 6. 小实验
不用代码，用纸笔追踪计算量就能体会。

设你要生成 4 个词，每步注意力代价约等于"当前序列长度"（简化模型）。

朴素重算法每步代价：
- 第 1 词：1（重跑长度 1）
- 第 2 词：2（重跑长度 2）
- 第 3 词：3
- 第 4 词：4
- 但注意朴素法每步是重跑**整段**，单步代价是长度的平方级，这里按 $\sum i^2 = 1+4+9+16 = 30$。

KV Cache 法每步代价（只算新词对全部 K 的一次点积，正比于长度）：
- $1 + 2 + 3 + 4 = 10$。

把 4 换成 100：朴素法约 $\frac{100^3}{3} \approx 333000$，缓存法约 $\frac{100^2}{2} = 5000$。差了 60 多倍。序列越长，差距越夸张。你能亲眼看到"三次方 vs 二次方"如何拉开鸿沟。

进一步可心算：缓存的显存随长度线性涨。如果一个 token 的 K+V 占 1 个单位，生成到第 100 词时缓存占 100 单位，第 1000 词时占 1000 单位——这就是长上下文为什么吃显存。

### 7. 批判性思考
局限性：
- **显存随上下文长度线性增长**，是超长上下文（如 128K token）推理的主要瓶颈。模型权重是固定的，但 KV Cache 会随对话变长而膨胀，最终撑爆显存。
- 高并发服务时，每个用户的对话各占一份 KV Cache，显存碎片化严重——这正是 PagedAttention（像操作系统管理虚拟内存那样分页管理缓存）要解决的问题。
- KV Cache 让单步计算变成"读一大块缓存 + 少量计算"，于是推理从计算瓶颈变成**显存带宽瓶颈**——瓶颈是把缓存从显存搬到计算单元的速度，而不是算力。

深度问题：既然 KV Cache 的显存正比于 $2 \times L \times n \times d_{model}$，如果让多个注意力头**共享同一份 K、V**（这正是 Multi-Query / Grouped-Query Attention 的思路），缓存能压缩几倍？代价是什么——共享 K、V 会不会削弱"多头从不同子空间看问题"的表达力？这个"速度/显存 vs 表达力"的权衡，你会怎么取舍？

### 8. 和后续知识的连接
- 直接承接**自回归生成**和**因果注意力**：KV Cache 之所以成立，全靠因果掩码保证旧词表示不变；它优化的正是自回归"一次一个词"的流程。
- 直接承接**缩放点积注意力**和**多头注意力**：缓存的就是每层每头的 K、V 矩阵，理解 QKV 的分工是理解"为何只缓存 KV"的前提。
- 向前连接到**高效注意力**（类目 19）：Multi-Query Attention、Grouped-Query Attention 本质都是"压缩 KV Cache"的方案；FlashAttention 优化的也是注意力的显存读写，和 KV Cache 的带宽瓶颈是同一类问题。
- 向前连接到**大模型应用与服务**（类目 20）：长上下文 RAG、多轮对话的工程实现，绕不开 KV Cache 的显存管理（PagedAttention、缓存复用、前缀缓存 prefix caching）。理解了 KV Cache，你才真正理解 LLM "为什么能边想边说，又为什么上下文越长越贵"。
