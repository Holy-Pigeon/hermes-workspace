## 今日知识点：Grouped-Query Attention（分组查询注意力，GQA）——在"每头一套钥匙"和"全员共用一套钥匙"之间找平衡

### 所属类目
类目 19：高效注意力

### 问题入口
我们之前讲过 KV Cache：自回归生成时，为了不每吐一个字就把前文重算一遍，模型会把每一层、每个注意力头算过的 Key 和 Value 缓存起来。这招把计算省下来了，但带来一个新瓶颈——**显存（以及显存带宽）**。

算一笔账：一个有 $L$ 层、每层 $H$ 个注意力头、每个头维度 $d_h$ 的模型，处理长度为 $n$ 的序列、批大小为 $b$ 时，KV Cache 要存的浮点数个数是

$$
2 \times b \times L \times H \times n \times d_h
$$

那个 $2$ 是因为 Key 和 Value 各存一份。以 LLaMA-2-70B 为例（$L=80$、$H=64$、$d_h=128$），生成到 $n=4096$、批大小 $b=1$、用 FP16（每数 2 字节）时，KV Cache 就要约 $2\times80\times64\times4096\times128\times2$ 字节 $\approx 10.7$ GB——而且这还只是 batch=1。批大小一上去、上下文一拉长，KV Cache 直接撑爆显存，并且每生成一个 token 都要把这一大坨从显存搬到计算单元，**带宽**成了真正的天花板。

问题来了：**能不能在几乎不掉精度的前提下，把 KV Cache 砍掉好几倍？**

### 朴素想法
最直接的念头：既然 Key/Value 这么占地方，那就**让所有注意力头共用同一套 Key 和 Value**，只保留每个头各自的 Query。这样 KV Cache 里 $H$ 个头的 K、V 塌缩成 1 套，显存直接除以 $H$。

这正是 2019 年提出的 **Multi-Query Attention（MQA，多查询注意力）**：$H$ 个 Query 头，但只有 1 个 Key 头、1 个 Value 头，大家抢着用。

### 卡住的地方
MQA 确实把 KV Cache 砍到了 $1/H$，推理飞快。但它走得太极端，暴露两个问题：

1. **精度掉点**：每个 Query 头本来可以"问不同的问题、查不同维度的信息"，现在所有头被迫面对同一套 Key/Value，表达能力被压缩，下游任务质量出现可观的下降，训练也更不稳定。
2. **并行尴尬**：在多卡张量并行时，注意力头通常按卡切分。MQA 只有一份 K/V，要么复制到每张卡（省不了多少），要么所有卡共享一份（通信开销）。

于是出现了一个经典的"过犹不及"困境：
- **MHA（标准多头）**：$H$ 套 K/V，质量最好，但 KV Cache 最大。
- **MQA（多查询）**：$1$ 套 K/V，最省，但质量掉、不好并行。

中间那条路在哪？

### 1. 概念
**Grouped-Query Attention（GQA）**：把 $H$ 个 Query 头分成 $G$ 个组，**同一组内的 Query 头共享一套 Key/Value**，不同组各有自己的 K/V。于是 K/V 的头数从 $H$ 降到 $G$（其中 $1 \leq G \leq H$）。

它把 MHA 和 MQA 统一成了一个连续光谱的两个端点：
- $G = H$：每个 Query 头独享一套 K/V，退化成 **标准 MHA**。
- $G = 1$：所有 Query 头共享一套 K/V，退化成 **MQA**。
- $1 < G < H$：就是 **GQA**，在质量和显存之间取折中。LLaMA-2-70B 用的是 $H=64$、$G=8$，即每 8 个 Query 头共享一套 K/V。

**通俗理解**：把注意力头想成一群侦探（Query），每个侦探都想破自己的案子。Key/Value 是案卷档案库。
- MHA 是每个侦探配一整套私人档案库——查得准，但租金（显存）爆炸。
- MQA 是全公司只有一个公共档案库——省钱，但所有人挤一个柜子，常常找不到自己要的卷宗。
- GQA 是把侦探分成几个小组，每组共享一个档案库——既省了大头租金，组内又能针对性地组织档案。

但类比到此为止，回到机制：这里"共享 K/V"的精确含义是——计算注意力时，组内每个 Query 头各自算自己的 $QK^\top$ 打分和加权，只不过它们点乘和加权的对象是**同一份** $K$、$V$ 矩阵。Query 仍然是 $H$ 个独立的投影，只有 K/V 的投影矩阵数量减少了。

### 2. 推导与证明
我们一步步看 GQA 在数学上改了什么、以及为什么能省。

设模型隐藏维度 $d$，头维度 $d_h = d/H$。

**关键步骤：**

1. **投影阶段的变化。** 标准 MHA 里，Query、Key、Value 各有 $H$ 套投影：
   $$
   Q_i = X W_i^Q,\quad K_i = X W_i^K,\quad V_i = X W_i^V,\quad i = 1,\dots,H
   $$
   其中 $X \in \mathbb{R}^{n \times d}$ 是输入，每个 $W_i^{\cdot} \in \mathbb{R}^{d \times d_h}$。GQA 保留 $H$ 套 $W_i^Q$，但只保留 $G$ 套 $W_g^K, W_g^V$（$g=1,\dots,G$）。这一步直接把 K/V 投影参数和它们的输出缓存量从 $H$ 份降到 $G$ 份。**为什么这么做**：缓存负担全在 K/V 上（Query 不进 KV Cache，因为当前 step 的 Query 用完即弃），所以只需对 K/V 动刀。

2. **分组映射。** 定义一个把 Query 头映射到组的函数 $\text{group}(i) = \lceil i / (H/G) \rceil$。第 $i$ 个 Query 头算注意力时，用的 Key/Value 是第 $g=\text{group}(i)$ 组的：
   $$
   \text{head}_i = \text{softmax}\!\left(\frac{Q_i K_g^\top}{\sqrt{d_h}}\right) V_g
   $$
   **为什么这么做**：这保证了 Query 的多样性（$H$ 个独立 $Q_i$）不丢，只在 K/V 上做共享，把质量损失控制在最小。

3. **缓存量的对比。** KV Cache 大小从
   $$
   2 \times b \times L \times H \times n \times d_h
   $$
   变成
   $$
   2 \times b \times L \times G \times n \times d_h
   $$
   缩小倍数正好是 $H / G$。LLaMA-2-70B 的 $H/G = 64/8 = 8$，KV Cache 直接小 8 倍——前面算的 10.7 GB 降到约 1.3 GB。**为什么重要**：自回归解码是**带宽受限（memory-bound）**的，每个 token 都要把整个 KV Cache 读一遍，缓存小 8 倍，意味着读取量小 8 倍，吞吐显著上升。

4. **如何从已有的 MHA 模型得到 GQA（uptraining）。** GQA 论文给了一个聪明的转换：把一个已训练好的 MHA 模型，每组内的 $H/G$ 个 K（和 V）投影做**均值池化**，得到该组共享的 K/V 投影，然后只用约 5% 的原始预训练算力继续训练（uptrain）一下，就能把质量恢复到接近 MHA。**为什么这么做**：均值池化是对"组内多个头本就相似"的一个合理初始化，比随机初始化收敛快得多，避免了从头训练的巨大成本。

证明的核心直觉不是某个等式，而是一个**容量-成本权衡的论证**：注意力质量主要由 Query 的多样性和 K/V 的表达力共同决定；实验发现 K/V 头数从 $H$ 降到一个适中的 $G$（如 8）时，质量曲线很平、几乎不掉，而成本曲线陡降——所以存在一个"甜点区"。

### 3. 来源与历史动机
- **MQA 论文**：《Fast Transformer Decoding: One Write-Head is All You Need》
- 作者：Noam Shazeer（Transformer/MoE 的核心作者之一）
- 时间：2019 年
- 链接：https://arxiv.org/abs/1911.02150

- **GQA 论文**：《GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints》
- 作者：Joshua Ainslie、James Lee-Thorp、Michiel de Jong 等（Google Research）
- 时间：2023 年（EMNLP 2023）
- 链接：https://arxiv.org/abs/2305.13245

历史背景：2019 年 MQA 提出时，问题是 Transformer 自回归解码太慢，瓶颈在反复加载 K/V。MQA 解决了速度，但质量代价让它没被广泛采用。直到 2023 年大模型推理成本成为产业级痛点，GQA 用"折中 + 从 MHA checkpoint 低成本转换"的双重设计击中要害，迅速成为 LLaMA-2、LLaMA-3、Mistral、Qwen 等主流开源大模型的默认注意力结构。可以说 GQA 是"工程现实倒逼架构设计"的典型案例。

### 4. 核心用途
用途一：**大模型推理提速与降本**。这是 GQA 的主战场。在长上下文、大批量服务场景下，KV Cache 是显存和带宽的头号消耗者，GQA 把它砍 4～8 倍，直接提升单卡能承载的并发请求数和生成吞吐——线上 LLM 服务（如聊天 API）几乎都受益于此。

用途二：**长上下文模型的可行性基础**。要支持 32K、128K 甚至更长的上下文，KV Cache 随 $n$ 线性增长会失控。GQA 把每 token 的缓存成本压下来，是长上下文模型能跑起来的前提之一（常和 FlashAttention、KV Cache 量化一起用）。

用途三：**老模型低成本升级**。已经花大价钱训好的 MHA 模型，不必推倒重来——用 uptraining 方法均值池化 + 5% 算力微调，就能转成推理更快的 GQA 版本，保护既有投资。

### 5. 反直觉点与常见误区
反直觉点：**GQA 减少的是 Key/Value 头，而不是 Query 头**，而且它几乎不省主要的矩阵乘法计算量（FLOPs 基本不变，Query 还是 $H$ 个）。它省的是**显存占用和显存带宽**。很多人第一反应以为 GQA 是"减少注意力头让计算变少"，错了——自回归解码的瓶颈根本不是算力，而是反复搬运 KV Cache 的带宽，GQA 正是冲着带宽去的。这与我们讲 FlashAttention 时的洞察一脉相承：**现代注意力优化，赢在 IO，不在 FLOPs。**

常见误区：
- 误区一："GQA 共享 K/V 会让组内的头算出一样的结果。" 错。组内每个头有**各自独立的 Query 投影** $W_i^Q$，所以打分 $Q_i K_g^\top$ 各不相同，注意力分布也不同，只是它们点乘和加权的 K/V 矩阵碰巧是同一份。
- 误区二："$G$ 越小越好，反正省得多。" 错。$G \to 1$ 就退化成 MQA，质量明显下滑。实践中 $G=8$ 是被反复验证的甜点，省得多又几乎不掉点；盲目压到 $G=1$ 或 $2$ 通常得不偿失。

### 6. 小实验
不用写复杂代码，用一张纸算"省了多少"，建立量化直觉。

设一个小模型：层数 $L=32$，Query 头数 $H=32$，头维度 $d_h=128$，序列长度 $n=2048$，批大小 $b=4$，FP16（2 字节/数）。

KV Cache 字节数 $= 2 \times b \times L \times (\text{KV头数}) \times n \times d_h \times 2$。

- **MHA**（KV 头数 $=32$）：$2\times4\times32\times32\times2048\times128\times2 \approx 8.6$ GB
- **GQA, $G=8$**（KV 头数 $=8$）：上式 KV 头数换成 8，结果 $\approx 2.1$ GB
- **MQA, $G=1$**（KV 头数 $=1$）：$\approx 0.27$ GB

你会直观看到：从 MHA 到 GQA($G$=8)，缓存掉到 $1/4$；再到 MQA 掉到 $1/32$。再结合"GQA 几乎不掉质量、MQA 会掉质量"这一实验事实，就能理解为什么产业界普遍停在 $G=8$ 而非一路压到 1——这就是"甜点"的含义。

进阶手算：把上面 GQA 的数字换成 LLaMA-2-70B 的配置（$L=80,H=64,G=8,d_h=128,n=4096,b=1$），验证一下正文里 10.7 GB→1.3 GB 那个 8 倍关系。

### 7. 批判性思考
局限性：
- GQA 是一个**经验折中**，$G$ 的最优值依赖模型规模和任务，没有理论上的"最佳分组数"，要靠实验扫。
- 它只压了 KV Cache 的**头数维度**，没动 $n$（序列长度）和 $d_h$。当上下文极长时，KV Cache 仍随 $n$ 线性膨胀——GQA 缓解但不根治长上下文的显存问题，还需配合 KV 量化、滑动窗口、缓存淘汰等手段。
- 从 MHA uptrain 到 GQA 虽便宜，但仍需额外算力和数据，且均值池化对某些任务可能损失特定头学到的精细模式。

深度问题：既然瓶颈是 KV Cache 的搬运带宽，除了"减少 K/V 头数"（GQA 的思路），还有哪些正交的攻击角度？（提示：可以从"每个数存几个比特"——KV Cache 量化，"是否每个 token 都要留"——缓存淘汰/滑动窗口，"能否把 K/V 压进低秩潜空间"——DeepSeek 的 Multi-head Latent Attention(MLA) 等方向想。GQA 砍的是头数这一维，这些方法砍的是其他维，彼此可叠加。）

### 8. 和后续知识的连接
- **直接承接**：本课直接建立在 **多头注意力**（类目 12）、**KV Cache**（类目 18）、**注意力计算复杂度**（类目 15）和 **FlashAttention**（类目 19）之上——理解了"KV Cache 为什么占显存、解码为什么是带宽受限"，才能看懂 GQA 在优化什么。
- **同一光谱**：GQA 把 MHA 和 MQA 统一成一个由 $G$ 控制的连续家族，这种"用一个超参数把两个极端连起来"的设计思想，在 Transformer 演化里反复出现。
- **向前延伸**：现代大模型的注意力优化是多管齐下的——GQA（减头数）+ FlashAttention（优化 IO）+ KV 量化（减比特）+ MLA（低秩压缩）常常组合使用。理解 GQA 是看懂 LLaMA、Mistral、Qwen 等当代开源模型架构图的必备一环，也是后续学习推理引擎（vLLM 的 PagedAttention 等）KV Cache 管理的基础。
